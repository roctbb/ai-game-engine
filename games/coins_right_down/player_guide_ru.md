# Монеты вправо и вниз
## Общий формат карты

Во всех задачах с полем карта передается как список столбцов: первая координата - `x`, вторая - `y`.

```python
width = len(board)
height = len(board[0]) if width else 0
cell = board[x][y]
```

`x` растет вправо, `y` растет вниз. Перед проверкой клетки всегда проверяйте границы: `0 <= x < width` и `0 <= y < height`.

Напишите функцию `make_move(x, y, board)`, которая возвращает `right` или `down`.

В функцию приходят:

- `x`, `y`: текущая позиция агента.
- `board`: матрица поля.

Значения в `board`:

- `-1` — стена.
- `0` — пустая клетка.
- `1` — монета.
- `2` — финиш.

Цель: дойти до финиша и собрать как можно больше монет. Карта каждый запуск генерируется случайно по `run_id`, но финиш всегда достижим, если аккуратно обходить стены. Поле стало больше: 10 на 10 клеток. На карте есть тупики: в них можно зайти, двигаясь только вправо и вниз, но потом нельзя добраться до финиша.

Полезная стратегия: заранее оценивать лучший путь до финиша с учетом монет, стен и тупиков. Если из клетки невозможно добраться до финиша, считайте такой путь очень плохим.

```python
def make_move(x, y, board):
    if x + 1 < len(board) and board[x + 1][y] != -1:
        return "right"
    return "down"
```

Для отладки можно использовать `print(...)`. Вывод появится в консоли под визуализацией синхронно с кадрами replay.

Идея динамического программирования:

1. Идите мысленно от финиша к старту.
2. Для каждой клетки храните, сколько монет можно собрать от нее до финиша.
3. Если справа стена, ход вправо запрещен.
4. Если снизу стена, ход вниз запрещен.
5. Если из соседней клетки нет пути к финишу, не выбирайте ее, даже если рядом кажется свободно.
6. Выберите направление, где будущий результат больше.

Так программа не просто бежит к финишу, а сравнивает варианты маршрута.

## Подробный разбор

### Что нужно написать

Это одиночная задача: один ваш алгоритм управляет попыткой от начала до конца.

Главный контракт: `Определите функцию make_move(x, y, board), возвращающую right/down, обходите стены и соберите максимум монет до финиша.`

Перед тем как писать сильную стратегию, убедитесь, что функция всегда возвращает действие правильного типа. Даже слабый, но стабильный бот лучше, чем бот, который иногда падает с ошибкой.

### Как читать карту и координаты

В задачах с картой используйте порядок координат `board[x][y]`: сначала столбец `x`, потом строка `y`. Это отличается от привычной записи в тетради, поэтому полезно сразу завести маленькие функции `in_bounds` и `is_safe`.

```python
DIRECTIONS = {
    "up": (0, -1),
    "right": (1, 0),
    "down": (0, 1),
    "left": (-1, 0),
}


def in_bounds(x, y, board):
    return 0 <= x < len(board) and 0 <= y < len(board[x])


def is_safe(x, y, board):
    return in_bounds(x, y, board) and board[x][y] not in (-3, -2, -1)
```


### Первый рабочий алгоритм

1. Для каждой клетки заведите значение: сколько монет можно собрать от нее до финиша.
2. Считайте значения от правого нижнего угла к левому верхнему.
3. Стены пропускайте: через них путь невозможен.
4. Тупики тоже должны получить очень плохую оценку, потому что из них нельзя дойти до финиша.
5. Для каждой клетки выбирайте максимум между ходом вправо и ходом вниз.
6. Во время игры делайте первый ход в сторону клетки с лучшим будущим результатом.

Для этой игры хорошая основная идея: динамическое программирование по клеткам: для каждой клетки храните лучший будущий результат до финиша.

### Как сделать решение сильнее

В одиночной задаче удобно улучшать решение постепенно: сначала безопасный бот, затем бот с поиском пути, затем бот, который учитывает очки, ресурсы и будущую опасность.

Полезный прием для отладки: печатайте короткие сообщения, например выбранную цель, оценку вариантов и итоговое действие. Тогда по replay легче понять, почему бот сделал именно этот ход.

### На что обратить внимание

Самая частая ошибка - выбрать красивую цель, но не проверить стену, ресурс или будущую опасность. В этой задаче особенно опасны тупики: клетка может быть свободной, но путь после нее заканчивается стенами.

Для первого самостоятельного улучшения попробуйте добиться, чтобы бот всегда возвращал разрешенную команду и не падал с ошибкой. После каждого изменения запускайте пример несколько раз: карты генерируются случайно, и стратегия не должна зависеть от одного заранее угаданного маршрута.

## Быстрый старт и простой пример

Эта игра - одиночная задача. Начните с простого решения, убедитесь, что оно запускается, а потом постепенно улучшайте стратегию.

### Контракт функции

Определите функцию make_move(x, y, board), возвращающую right/down, обходите стены и соберите максимум монет до финиша.

Координаты в играх с сеткой читаются одинаково: `x` растет вправо, `y` растет вниз. Если поле называется `board` или `maze`, карта хранится как список столбцов: `len(board)` — ширина, `len(board[0])` — высота, клетку читают как `board[x][y]` или `maze[x][y]`. Если в игре используется `state`, внутри него обычно есть текущая позиция, карта и цель.

### Минимальный пример

Этот пример специально простой. Он показывает формат функции и ответа, но не пытается быть сильной стратегией.

```python
def make_move(x, y, board):
    goal_x = len(board) - 1
    goal_y = len(board[0]) - 1

    if x < goal_x and board[x + 1][y] != -1:
        return "right"
    if y < goal_y and board[x][y + 1] != -1:
        return "down"
    return "right"
```

### Как улучшать решение

1. Сначала выпишите, какие данные реально приходят в функцию: координаты, карта, ресурс, счет, роль или `state`.
2. Сделайте проверку допустимых ответов: бот не должен ходить в стену, открывать уже открытую клетку или строить башню без бюджета.
3. Для задач с картой заведите маленькие функции проверки границ и чтения клетки через `board[x][y]`.
4. Если есть ключи, время, вода, огонь, батарейка, ящики или соперник, храните это в состоянии алгоритма, а не только координаты.
5. Сравните свое решение с примером из `examples`, запустите несколько случайных карт и посмотрите replay.

### Где посмотреть полный пример

- `examples/agent_coin_collector.py` для слота `agent`: Демо: максимум монет.

Полный пример можно открыть в папке `examples` рядом с этой документацией. Его удобно запускать как рабочую основу, а затем упрощать или усиливать под свою стратегию.


## Ошибки и подсказки

Если бот возвращает неправильную команду, идет в стену или ему не хватает ресурса, движок добавляет событие с полем `message`. Это короткое объяснение можно показывать участнику в интерфейсе.
