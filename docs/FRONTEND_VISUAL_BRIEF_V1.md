## 2. Визуальная цель продукта
Платформа должна восприниматься не как сухая админка и не как "детская игра", а как:
- учебная соревновательная лаборатория;
- место, где пишут код, тестируют стратегии и смотрят живые матчи;
- современный и собранный инструмент для участника и преподавателя.

Ключевой образ: **соревновательная лаборатория алгоритмов**.

Ассоциации, на которые нужно опираться:
- чистота;
- ясность;
- дисциплина;
- энергия соревнования;
- ощущение живого процесса;
- уважение к коду как к основному рабочему материалу.

## 3. Дизайн-принципы

### 3.1 Общие принципы
- интерфейс должен быть светлым по умолчанию;
- основной упор на читаемость, структуру и ясность действий;
- визуализация матчей и задач должна быть эмоциональным центром платформы;
- кодовый редактор должен ощущаться как полноценное рабочее место;
- у каждого режима платформы должен быть свой характер, но в рамках одной дизайн-системы.

### 3.2 Чего делать нельзя
- не строить интерфейс как CRUD-админку с одинаковыми таблицами;
- не делать безликий "обычный bootstrap";
- не превращать платформу в темный gamer-dashboard по умолчанию;
- не использовать случайные яркие акценты без семантики;
- не смешивать учебные, игровые и турнирные сценарии в одинаковые визуальные шаблоны.

### 3.3 Главный UX-принцип
На каждом экране должно быть очевидно:
- где я нахожусь;
- что является главным объектом страницы;
- какое действие здесь основное;
- что происходит прямо сейчас.

## 4. Визуальные режимы платформы
Платформа должна использовать одну общую дизайн-систему, но три визуальных режима:

1. `learning`
Сценарии:
- каталог задач;
- запуск `single_task`;
- история попыток;
- прогресс по темам.

Характер:
- спокойный;
- сосредоточенный;
- больше поясняющего текста;
- меньше давления по статусам.

2. `play`
Сценарии:
- команды;
- тренировочные лобби;
- ожидание матча;
- live-сессии.

Характер:
- более живой;
- больше статусов;
- больше realtime-индикаторов;
- больше плотности событий.

3. `competition`
Сценарии:
- турниры;
- сетка;
- результаты;
- санкции;
- spectator mode.

Характер:
- более строгий;
- собранный;
- акцент на итогах, местах и progression;
- меньше декоративности, больше иерархии и контроля.

4. `administration`
Сценарии:
- git sync;
- warnings;
- управление играми;
- runtime profiles.

Характер:
- более утилитарный;
- спокойный;
- высокий приоритет ошибок, статусов и журналов.

## 5. Цветовая система

### 5.1 Базовая палитра
Рекомендуемая базовая палитра:

- `--color-bg`: `#F5F1E8`
- `--color-surface`: `#FFFDF8`
- `--color-surface-2`: `#F2ECE1`
- `--color-text`: `#1F2430`
- `--color-text-muted`: `#667085`
- `--color-border`: `#D8D2C6`
- `--color-primary`: `#D9481C`
- `--color-primary-hover`: `#B93815`
- `--color-secondary`: `#0F7B8C`
- `--color-secondary-hover`: `#0B6270`
- `--color-success`: `#2E7D32`
- `--color-warning`: `#C58B00`
- `--color-danger`: `#B42318`
- `--color-info`: `#1D4ED8`

### 5.2 Семантика цветов
- `primary` используется для основного действия на экране;
- `secondary` используется для live/realtime, viewer, streaming, telemetry;
- `success` используется для `ready`, победы, успешного запуска;
- `warning` используется для `blocked`, спорных состояний, предупреждений;
- `danger` используется для ошибок, банов, destructive-actions;
- `muted` используется для вторичной информации и archived-элементов.

### 5.3 Градиенты и атмосфера
Платформа не должна опираться на плоский фон одного цвета.

Разрешены:
- мягкие радиальные градиенты на страницах каталога и турниров;
- тонкие шумовые или сеточные текстуры;
- subtle highlights вокруг viewer или hero-блоков.

Не использовать:
- агрессивные неоновые градиенты;
- фиолетовый как основной бренд-цвет;
- тяжелые dark-only стили для всей системы.

## 6. Типографика

### 6.1 Шрифтовая система
Рекомендуемый набор:
- интерфейсный текст: `IBM Plex Sans`
- выразительные заголовки: `Space Grotesk`
- код: `JetBrains Mono`

### 6.2 Иерархия
- `Display`: для названия больших игровых страниц и турниров
- `H1`: название страницы
- `H2`: основной блок страницы
- `H3`: секции и панели
- `Body`: основной интерфейсный текст
- `Small`: подписи, metadata, hints
- `Mono`: код, id, статусы запусков, snapshot refs

### 6.3 Тон текста
Текст должен быть:
- коротким;
- прямым;
- уверенным;
- без канцелярита;
- с русской локализацией по умолчанию.

## 7. Пространство, форма и тени

### 7.1 Скругления
- основные контейнеры: `16px`
- карточки: `18px`
- маленькие pills/status badges: `999px`
- редактор и viewer panels: `20px`

### 7.2 Отступы
Базовая сетка:
- шаг `4px`
- основной рабочий шаг `8px`
- секционные отступы `24px`, `32px`, `48px`

### 7.3 Тени
Тени должны быть:
- мягкими;
- теплыми;
- почти бумажными, а не "веб 2016".

Пример:
- `shadow-sm`: мягкая тень для card
- `shadow-md`: для floating panel
- `shadow-lg`: для modal, viewer shell, tournament spotlight

## 8. Shell приложения

### 8.1 Основной layout desktop
На desktop приложение должно использовать трехуровневую структуру:

1. `AppSidebar`
Содержит:
- logo;
- основные разделы;
- быстрый переход к активным лобби/турнирам;
- профиль пользователя.

2. `TopBar`
Содержит:
- breadcrumb;
- контекст страницы;
- быстрые действия;
- глобальный поиск в будущем.

3. `ContentArea`
Содержит:
- основной контент;
- при необходимости правую contextual panel.

### 8.2 Layout mobile
На mobile:
- sidebar заменяется drawer;
- появляется bottom navigation для ключевых разделов;
- editor/viewer/panels переключаются вкладками;
- турнирная сетка и viewer открываются во весь экран.

### 8.3 Главные разделы навигации
Верхнее меню должно быть коротким:

1. `Задачи`
   - каталог задач;
   - запуск и история попыток;
   - прогресс и сравнение с другими решениями.

2. `Лобби и соревнования`
   - список открытых лобби;
   - вход в лобби;
   - вкладки лобби `Код`, `Лобби`, `Игра`, `Соревнование`, `Архив соревнований`;
   - teacher/admin видит `+` для создания лобби.

3. `Игры`
   - список игр;
   - документация игр;
   - teacher/admin видит `+` для добавления игры и действия настройки игр.

4. `Статус системы / общие настройки`
   - виден только admin;
   - состояние сервисов, workers, game sources, runtime/system settings.

Не выносить в главное меню для ученика:
- `Команды`;
- `Матчи`;
- `Повторы`;
- общее `Администрирование`.

Эти сущности должны открываться из контекста задач, лобби, игр или системного раздела.

## 9. Bootstrap 5 theme strategy
Bootstrap 5 должен использоваться как техническая база, а не как визуальный язык по умолчанию.

Требования:
- задать собственные CSS variables поверх Bootstrap;
- переопределить buttons, cards, forms, navs, dropdowns, modals;
- не использовать стандартный внешний вид `.card`, `.btn`, `.table` без кастомизации;
- формы должны быть более воздушными и современными, чем bootstrap default.

## 10. Основные паттерны экранов

### 10.1 Игры
Раздел `Игры` должен выглядеть как витрина модулей платформы и точка входа в документацию игр, а не как список ссылок.

Каждая карточка игры должна содержать:
- название;
- краткое описание;
- тип режима;
- размер команды;
- spectator mode / leaderboard flags;
- ссылку на документацию игры;
- основное действие.

Фильтры:
- по режиму игры;
- по темам для `single_task`;
- по сложности;
- по статусу доступности.

### 10.2 Каталог задач
Экран задач должен визуально отличаться от каталога игр.

Обязательные элементы:
- hero-блок с объяснением режима обучения;
- фильтры по темам и сложности;
- группировка задач по темам и сложности;
- карточки задач внутри групп с процентом успеха или числом решивших;
- блок "решено больше всего задач" или аналогичный summary.

Каталог задач не должен быть плоской сеткой по умолчанию. Основное представление должно помогать пользователю увидеть учебную карту:
- тема как крупный раздел;
- сложность как подгруппа, колонка, таб или компактный раздел внутри темы;
- прогресс по теме и сложности рядом с группой;
- активный поиск или фильтр может временно превращать выдачу в плоский список результатов.

Режим задач должен мотивировать через конкуренцию и видимый результат:
- пользователь должен видеть, как именно сработал его код через viewer/replay;
- рядом с результатом нужно показывать сравнение с другими пользователями, если данные доступны;
- если есть score, показывать место, лучший score, percentile или близкую понятную метрику;
- если score нет, показывать число решивших, успешность, историю собственных попыток и честное сравнение по доступным данным.

### 10.3 Код игрока
Это один из важнейших views платформы, но для ученика он обычно встроен в задачу или вкладку `Код` лобби.

На текущем этапе это не экран выбора команды. Для пользователя игрок - это canonical набор кодов для конкретной игры, один на пользователя и игру. Если игра требует несколько ролей, workspace показывает несколько слотов; если один агент - один слот.

Структура desktop:
- левая колонка: слоты/роли команды;
- центральная колонка: кодовый редактор;
- правая колонка: шаблон, лог, статус snapshot, подсказки.

Обязательные состояния слота:
- заполнен;
- пуст;
- обязателен;
- изменен после последнего snapshot;
- заблокирован политикой соревнования.

Основной акцент:
- пользователь всегда должен понимать текущее состояние канонического кода команды;
- интерфейс может показывать только preview того, какие snapshot будут зафиксированы, если запуск создать сейчас.
- если пользователь редактирует код во время ожидания или текущей игры, UI должен явно говорить, что сохранения применятся к следующей игре.

### 10.4 Single-task run page
Страница запуска `single_task` должна быть гибридом viewer и IDE.

При этом это не второй независимый редактор:
- экран обязан работать поверх того же canonical workspace однослотовой персональной команды пользователя;
- source of truth для кода остается `TeamSlotCode`;
- run page - это execution shell над тем же workspace.

Структура:
- viewer / игровой мир;
- редактор;
- actions panel с `Запустить`, `Остановить`, `Сбросить к шаблону`;
- history panel;
- result panel;
- debug log panel.

Приоритеты:
- viewer должен быть крупным;
- запуск должен быть главным CTA;
- ошибки должны быть легко читаемыми;
- история попыток должна ощущаться как timeline прогресса, а не как скучная таблица.
- блок сравнения с другими решениями должен быть рядом с result/replay, чтобы пользователь понимал не только "работает ли код", но и "насколько хорошо".

### 10.5 Тренировочное лобби
Лобби должно ощущаться как основной рабочий экран тренировки: здесь пользователь пишет код, нажимает `Готов` / `Play`, видит очередь, смотрит свою текущую игру и может участвовать в соревновании, прикрепленном к этому лобби.

Для `small_match` и `massive_lobby` используется один и тот же UI лобби. Отличается только политика попадания в игру:
- в `small_match` готовый игрок может ждать в очереди, пока матчмейкинг подберет состав;
- в `massive_lobby` все готовые игроки могут сразу попасть в текущую игру/мир.

Это различие должно объясняться статусом игрока и коротким текстом политики лобби, а не разными экранами.

Обязательные блоки:
- заголовок лобби;
- игра, версия, тип доступа, публичность;
- участники по группам: `играют`, `готовятся`, `в очереди`;
- понятная статистика участников: победы, средние очки, недавняя форма или другой доступный показатель силы;
- статус пользователя в этом лобби;
- текущие матчи;
- архив матчей этого лобби;
- текущая игра, если матч найден;
- активное соревнование, если teacher/admin запустил его в этом лобби;
- архив соревнований, если в лобби уже были завершенные соревнования;
- настройки лобби;
- код доступа, если лобби закрытое.

Для student-flow используется трехвкладочная структура:
- `Код` - заполнить роли, сохранить изменения и открыть документацию игры/задачи;
- `Лобби` - увидеть группы участников, очередь, текущие матчи и архив;
- `Игра` - смотреть текущую пошаговую игру пользователя.

Документация игры/задачи является частью задачи, а не внешней справкой. На вкладке `Код` ссылка или панель документации должна быть заметной, но не конкурировать с редактором. Если игровой пакет поставляет player guide и SDK guide, UI должен дать быстрый доступ к обоим.

Дополнительные вкладки появляются по состоянию:
- `Соревнование` - пока идет активное соревнование или оно ожидает ручного завершения teacher/admin;
- `Архив соревнований` - после первого завершенного соревнования в лобби.

Во время соревнования:
- лобби показывает persistent banner `Идет соревнование`;
- вход новых участников и выход текущих участников заблокированы;
- вкладка `Соревнование` показывает realtime bracket/rounds view;
- матч в сетке кликабелен и открывает match viewer;
- текущая игра пользователя все равно отображается во вкладке `Игра`.

Игроки в лобби должны визуально различаться по состояниям:
- `playing`
- `preparing`
- `queued`
- `blocked`

Карточка или строка игрока должна помогать быстро понять, с кем пользователь играет и насколько хорошо этот игрок выступает. Статистика должна быть компактной и сравнимой между участниками.

На вкладке `Игра` основной layout должен быть двухколоночным:
- слева визуализация игры;
- справа статистика игроков, счет, метрики и текущий статус.

Если пользователь меняет код во время ожидания или текущей игры и нажимает `Готов` / `Play`, UI должен явно показывать, что изменения применятся со следующей игры.

Teacher/admin actions в лобби:
- `Начать соревнование` - открывает выбор настроек и запускает соревнование в этом лобби;
- `Завершить соревнование` - доступно после завершения всех игр и выбора победителя, скрывает активную вкладку `Соревнование` и оставляет итоги в `Архиве соревнований`.

### 10.6 Соревнование
Соревнование является главным event-view внутри лобби.

Для основного student-flow это вкладка `Соревнование` на странице лобби. Отдельная страница соревнования допустима как deep-link или teacher/admin view, но не должна быть обязательным маршрутом для ученика.

Для `single_elimination`:
- bracket view только там, где он не искажает модель турнира;
- список матчей раунда;
- standings/metadata sidebar.

Будущие `round_robin` и `swiss` не входят в текущий v2 UI. После появления backend-поддержки для них потребуются:
- standings table;
- список туров;
- матчевые карточки.

Для elimination-матчей с `N > 2` командами или `top-k` продвижением:
- UI обязан поддерживать multi-team match nodes;
- если bracket становится нечитабельным, главным представлением должен быть rounds/table view, а не бинарная сетка.

Клик по матчу должен:
- открывать viewer;
- или открывать side panel с деталями и кнопкой перехода.

### 10.7 Просмотр матча
Экран просмотра должен быть самым эмоционально сильным интерфейсом платформы.

Обязательные элементы:
- крупный renderer;
- overlay со счетом и статусом;
- timeline;
- event feed;
- stats panel;
- переключение `live / replay`.

Публичный spectator mode должен работать без ощущения "урезанной версии".

### 10.8 Экран git sync и администрирования
Admin-экраны должны быть простыми и строгими.

Обязательные элементы:
- список game sources;
- commit SHA и status sync;
- validation results;
- build logs;
- warnings / security issues;
- явные destructive confirmations.

## 12. Система статусов

### 12.1 Общие статусы
Для всех ключевых сущностей нужны единообразные визуальные паттерны:
- `draft`
- `ready`
- `running`
- `paused`
- `completed`
- `finished`
- `failed`
- `blocked`
- `canceled`

### 12.2 Цветовая карта статусов
- `ready`: зеленый
- `running`: бирюзовый или сине-зеленый
- `paused`: приглушенный янтарный
- `blocked`: янтарный
- `failed`: красный
- `canceled`: серо-красный
- `draft`: нейтральный серо-песочный

### 12.3 Realtime-поведение
Статусы live-объектов должны сопровождаться:
- subtle pulse;
- мягким обновлением текста;
- без резких миганий;
- без агрессивных loader-анимаций.

## 13. Motion и анимации

### 13.1 Общие правила
Анимации должны:
- помогать понять изменение состояния;
- не мешать чтению;
- не замедлять взаимодействие.

### 13.2 Допустимые анимации
- page-enter fade/slide;
- staggered reveal карточек;
- подсветка обновившегося матча;
- smooth expand/collapse у side panels;
- мягкое появление timeline scrubber.

### 13.3 Недопустимые анимации
- bounce;
- heavy parallax;
- бесконечно мигающие индикаторы;
- анимации ради анимации.

## 14. Доступность
Минимальные требования:
- контраст текста соответствует WCAG AA;
- все интерактивные элементы доступны с клавиатуры;
- editor, buttons, tabs, modals и bracket nodes имеют фокус-состояния;
- viewer controls доступны без мыши;
- цвет не является единственным носителем статуса.

## 15. Renderer shell и iframe contract
Renderer сторонней игры живет внутри iframe, но shell платформы отвечает за UX.

Shell обязан:
- показывать loading/skeleton;
- обрабатывать ошибки renderer;
- уметь перезапустить renderer;
- держать layout viewer стабильным;
- прокидывать только безопасные данные через `postMessage`.

Renderer не должен:
- ломать layout страницы;
- напрямую влиять на глобальный shell;
- требовать прямого доступа к auth token.

## 16. Правила для генерации Vue 3 frontend

### 16.1 Архитектура frontend
Рекомендуемая структура:

```text
frontend/
  src/
    app/
    layouts/
    pages/
    widgets/
    features/
    entities/
    shared/
      ui/
      lib/
      api/
      config/
      styles/
```

Допустимы и другие варианты, если сохраняются:
- модульность;
- переиспользуемость;
- отделение доменных компонентов от инфраструктурных;
- отсутствие giant-page components.

### 16.2 Styling approach
Рекомендуется:
- CSS variables;
- Bootstrap utility layer;
- локальные компонентные стили;
- единый theme entrypoint.

Не рекомендуется:
- inline-style как основной способ верстки;
- неструктурированные глобальные CSS файлы;
- тяжелая кастомизация через `!important`.

### 16.3 Работа с Monaco
Редактор должен быть интегрирован как first-class citizen:
- светлая тема по умолчанию;
- аккуратный контейнер;
- сохранение состояния вкладок;
- понятный toolbar;
- отображение template/ref и runtime hints.

## 19. Инструкция для будущей кодогенерации frontend
При генерации frontend-кода обязательно:
- не использовать дефолтный внешний вид Bootstrap без кастомизации;
- строить UI вокруг domain objects платформы;
- закладывать responsive layout сразу;
- делать editor и viewer центральными объектами соответствующих экранов;
- использовать единые токены цветов, типографики, отступов и статусов;
- поддерживать визуальную консистентность между `learning`, `play`, `competition`, `administration`.

Если нужно выбирать между:
- "быстро и безлико"
- и "чуть дольше, но системно и выразительно",

выбирать второй вариант.
