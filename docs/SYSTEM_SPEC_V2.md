## 2. Цель продукта
Платформа предназначена для:
- обучения программированию через интерактивные визуализируемые задачи;
- проведения тренировочных матчей и соревнований между ботами;
- поддержки командных и индивидуальных сценариев;
- запуска пользовательского кода в изолированной инфраструктуре;
- подключения новых игр из репозитория по стандартному протоколу.

## 3. Основные продуктовые режимы

### 3.1 `single_task`
Одиночная задача, в которой участвует один пользователь или одна команда из одного слота.

Примеры:
- выход из лабиринта;
- сбор монет при движении по карте вправо и вниз;
- расстановка башен для защиты базы;
- задачи в стиле CodeCombat через игровой SDK.

Свойства:
- безлимитное число попыток;
- запрещен параллельный запуск двух решений одним и тем же пользователем по всей платформе;
- следующий запуск возможен только после завершения или явной остановки предыдущего;
- результаты, история попыток, реплеи и отладочные логи сохраняются;
- лидерборд может быть включен или выключен для конкретной игры;
- если у игры нет score-модели, платформа может показывать только список успешно решивших.

### 3.2 `small_match`
Матчи с небольшим числом участников или команд.

Примеры:
- крестики-нолики до 5 на большой доске;
- дуэльные игры;
- игры 3v3 или free-for-all на ограниченное число участников.

Свойства:
- есть тренировочные лобби;
- есть соревнования;
- состав матча может быть больше 2 участников;
- продвижение по турниру настраивается;
- игра может возвращать очки, а турнир использует правила интерпретации результатов.

### 3.3 `massive_lobby`
Игры, где одновременно играет большое число команд.

Пример:
- танки.

Свойства:
- тренировочные лобби живут, пока их не закроет teacher/admin;
- в тренировке разрешен вход и выход;
- в соревновании вход и выход запрещены после фиксации состава;
- код, измененный во время текущего запуска, применяется только к следующему матчу или следующему новому запуску игры;
- сама игра знает, запущена она в режиме тренировки или соревнования.

## 4. Ключевые архитектурные решения

### 4.1 Платформа должна поддерживать три режима игр
Поддержка всех трех режимов обязательна:
- `single_task`;
- `small_match`;
- `massive_lobby`.

### 4.2 Игра - основная единица интеграции
На первом этапе каждая задача считается отдельной игрой.

В будущем архитектура должна позволять одной игре иметь несколько сценариев или уровней, но в `v1/v2` обязательным минимальным контрактом является подключение новой логики как новой игры.

### 4.3 Команды обязательны
Система должна поддерживать команды.

MVP-модель команды:
- команда создается для конкретной игры;
- у команды есть капитан;
- пока только капитан редактирует коды;
- команда содержит набор кодов по слотам/ролям, определенным игрой;
- команда присоединяется к лобби целиком;
- одна и та же команда может одновременно участвовать в нескольких лобби, если у нее есть доступ;
- в одном конкретном лобби у капитана не может быть одновременно более одной команды.

### 4.4 Пользовательский код поддерживает два режима API
Платформа должна поддерживать два режима взаимодействия пользовательского кода с игрой:
- `turn_based` - игра вызывает код пошагово;
- `script_based` - пользователь пишет более "сценарный" код через игровой SDK.

Оба режима обязаны работать поверх одного общего игрового рантайма и одного общего процесса оркестрации запусков.

### 4.5 Git-интеграция обязательна
Игры могут храниться:
- в монорепозитории платформы;
- во внешнем публичном git-репозитории, который укажет admin/teacher.

Подключение внешней игры происходит вручную через UI.

### 4.6 Docker-only execution
Весь пользовательский код и код игрового движка выполняются только в Docker.

### 4.7 Масштабирование через удаленные игровые воркеры
Игровые воркеры должны выноситься на отдельные серверы.

### 4.8 Frontend - полноценный SPA
Frontend должен быть переписан как SPA на Vue 3.

### 4.9 Renderer игры загружается гибридно
Платформа предоставляет общую оболочку SPA, а сама игра может поставлять свой renderer.

Для безопасности renderer из внешнего git-репозитория загружается в `iframe`.

## 5. Технологический стек (фиксированный для целевой реализации)

### 5.1 Backend
- Python 3.12+
- FastAPI
- SQLAlchemy 2.x
- Alembic
- Pydantic v2
- Redis
- WebSocket endpoints для realtime

### 5.2 Frontend
- Vue 3
- Vite
- TypeScript
- Pinia
- Vue Router
- Bootstrap 5
- CodeMirror 6 (web code editor with syntax highlight)

### 5.3 Infra
- PostgreSQL 15+
- Redis 7+
- Docker
- docker compose / compose.yml
- Nginx как reverse proxy для production

### 5.4 Почему сохраняется Redis
Redis остается в системе, потому что:
- уже присутствует в текущем проекте;
- подходит для очередей, pub/sub, блокировок, heartbeat и transient-state;
- на текущем этапе его преимуществ достаточно, чтобы не добавлять более тяжелую шину сообщений.

### 5.5 Зафиксированные технологические решения `v2`
Следующие технологические решения считаются согласованными и не должны пересматриваться без отдельной причины:

1. Новый backend строится на `FastAPI`, а не на `Flask`.
2. Realtime transport между backend и SPA строится на обычном `WebSocket`, а не на `Socket.IO`.
3. `Redis` остается coordination layer для очередей, heartbeat, pub/sub и transient-state.
4. Планирование запусков реализуется собственным `scheduler-service`, а не через `Celery`.
5. Сборка игровых пакетов и исполнение игр разделяются на разные сервисы:
   - `builder-service`
   - `game-worker`
6. Внутренняя авторизация веб-приложения после GeekClass login строится на `server-side session` и cookie-based session model.
7. Тестовый стек фиксируется так:
   - backend: `pytest`, `pytest-asyncio`, `httpx`
   - frontend: `vitest`
   - e2e: `playwright`

### 5.6 Что сознательно не используется в `v2`
В `v2` не планируется использовать как базовую часть архитектуры:
- `Socket.IO`
- `Celery`
- `RabbitMQ`
- `Kafka`
- `GraphQL`
- `Kubernetes`

Если эти технологии когда-либо появятся позже, это должно быть оформлено как отдельное архитектурное решение, а не как "тихая" подмена стека.

## 6. Качество реализации и обязательные инженерные требования

### 6.1 DDD обязателен
Код должен строиться по DDD-подходу.

Требования:
- разделение на bounded contexts;
- доменная логика находится в `domain`, а не в роутерах и не в ORM-сущностях контроллерного слоя;
- application layer оркестрирует use-case'ы;
- infrastructure layer содержит БД, git, Docker, Redis, websocket transport, внешние интеграции;
- явные domain entities, value objects, repositories, services;
- инварианты агрегатов реализуются в доменной модели, а не размываются по коду.

### 6.2 Слои backend
Для каждого bounded context использовать структуру вида:

```text
backend/src/<context_name>/
  domain/
  application/
  infrastructure/
  api/
```

Общий слой:

```text
backend/src/shared/
  kernel/
  db/
  events/
  security/
  utils/
```

### 6.3 Правила чистоты кода
- строгая типизация в Python;
- без "god services" и без гигантских файлов;
- без бизнес-логики в HTTP endpoints;
- без прямого смешивания ORM-моделей и DTO фронтенда;
- без молчаливых `except:` в бизнес-коде;
- без скрытых side effect в моделях;
- все важные переходы состояния должны быть явно отражены в коде и тестах.

### 6.4 Обязательные тесты
Тесты обязательны.

Минимальный набор:
- unit tests для доменной логики;
- integration tests для БД, репозиториев, worker orchestration, git sync, Docker runner;
- API tests;
- contract tests для игровых пакетов;
- e2e smoke tests для ключевых UI-сценариев.

### 6.5 Definition of Done для любой новой фичи
Фича считается выполненной только если:
- есть код;
- есть тесты;
- соблюдены инварианты домена;
- обновлена OpenAPI-схема, если менялся API;
- обновлены пользовательские ошибки и статусы;
- есть логирование для критичных путей;
- нет регрессии базовых сценариев.

## 7. Bounded Contexts

Система разбивается минимум на следующие bounded contexts:

1. `identity`
2. `game_catalog`
3. `team_workspace`
4. `training_lobby`
5. `competition`
6. `execution`
7. `spectator_replay`
8. `antiplagiarism`
9. `administration`

### 7.1 `identity`
Отвечает за:
- логин через GeekClass;
- dev-login;
- локальный профиль пользователя;
- role checks.

### 7.2 `game_catalog`
Отвечает за:
- реестр игр;
- git-источники игр;
- версии игр;
- синхронизацию из git;
- метаданные игр;
- runtime profiles и доступные библиотеки;
- renderer assets.

### 7.3 `team_workspace`
Отвечает за:
- команды;
- слоты команды;
- коды по слотам;
- снимки кодов для матчей;
- шаблоны кода.

### 7.4 `training_lobby`
Отвечает за:
- тренировочные лобби;
- вход по открытому доступу или по коду;
- ready-state;
- очередь;
- автозапуск матчей;
- lifecycle тренировочных матчей.

### 7.5 `competition`
Отвечает за:
- соревнования;
- формат турнира;
- фиксацию состава;
- матчи соревнования;
- продвижение победителей;
- санкции;
- проверку схожести решений в рамках соревнования.

### 7.6 `execution`
Отвечает за:
- игровые воркеры;
- Docker runner;
- job scheduling;
- heartbeat;
- realtime events от движков;
- технический lifecycle запуска.

### 7.7 `spectator_replay`
Отвечает за:
- live-view матчей;
- хранение реплеев;
- публичный spectator mode;
- выдачу данных renderer'у.

### 7.8 `antiplagiarism`
Отвечает за:
- сравнение решений в рамках соревнования;
- формирование warning для teacher/admin;
- хранение suspicious pairs.

### 7.9 `administration`
Отвечает за:
- системные настройки;
- аудит;
- ручную модерацию;
- блокировки и баны.

## 8. Роли и права

### 8.1 `student`
Может:
- входить в систему;
- создавать команды для доступных игр;
- редактировать код своей команды, если он капитан;
- входить в открытые лобби;
- входить в лобби по коду;
- становиться `ready`;
- запускать одиночные задачи;
- смотреть свои результаты;
- смотреть публичные live/replay, если это разрешено игрой и лобби.

### 8.2 `teacher`
Может все, что `student`, и дополнительно:
- создавать и закрывать лобби;
- создавать соревнования;
- синхронизировать игры из git;
- включать и выключать публичный spectator mode;
- смотреть warnings по схожести решений;
- переводить участника/команду в `not_ready`;
- временно банить участника/команду;
- управлять доступом по коду;
- просматривать технические логи матчей;
- останавливать активные тренировочные матчи и лобби.

### 8.3 `admin`
Может все, что `teacher`, и дополнительно:
- управлять системными настройками;
- управлять runtime profiles;
- управлять глобальными параметрами безопасности;
- удалять игры и git-источники;
- просматривать и настраивать воркеры.

### 8.4 Источник прав
Роль пользователя берется из GeekClass JWT как есть.

Дополнительного локального маппинга ролей в `v2` не требуется.

## 9. Ubiquitous Language

### 9.1 Игра (`Game`)
Логическая игра, доступная на платформе.

### 9.2 Версия игры (`GameVersion`)
Неизменяемый снимок игры, полученный из конкретного commit SHA или из монорепозитория.

### 9.3 Источник игры (`GameSource`)
Запись о том, откуда подтягивается игра:
- `embedded`;
- `git`.

### 9.4 Команда (`Team`)
Глобальная сущность для конкретной игры.

### 9.5 Слот команды (`TeamSlot`)
Кодовое место внутри команды, определяемое игрой.

### 9.6 Снимок кода (`CodeSnapshot`)
Неизменяемая версия кода, реально использованная в матче или попытке.

### 9.7 Тренировочное лобби (`TrainingLobby`)
Долгоживущее лобби, в которое команды могут входить, выходить и становиться готовыми.

### 9.8 Соревнование (`Competition`)
Событие, в котором состав фиксируется, применяются правила турнира и санкции.

### 9.9 Матч (`Match`)
Конкретный игровой запуск для двух или более команд.

### 9.10 Попытка (`SingleTaskAttempt`)
Запуск одиночной игры пользователем.

### 9.11 Запуск (`Run`)
Техническая единица исполнения в worker plane.

### 9.12 Реплей (`Replay`)
Набор кадров, событий и итоговой информации для просмотра после завершения.

## 11. Инварианты домена

### 11.1 Игра
- у игры всегда есть активная версия;
- игра не может быть доступна пользователям без валидной версии;
- игра не может быть синхронизирована во время активного соревнования по этой игре;
- если у игры есть активные тренировочные лобби, синхронизация с несовместимым изменением схемы обязательных слотов запрещена.

### 11.2 Команда
- команда принадлежит ровно одной игре;
- слот команды должен соответствовать слотам, объявленным активной версией игры;
- команда не может перейти в `ready`, если не заполнены все обязательные слоты.

### 11.3 Тренировочное лобби
- teacher/admin - единственные, кто может создавать лобби;
- команда может войти в лобби даже без полного набора кодов;
- команда не может стать `ready`, пока обязательные слоты пусты;
- если код изменился до фактического запуска матча, используется новый код;
- лобби живет, пока его не закроет teacher/admin;
- при обновлении игры все команды в `ready` обязаны пройти повторную валидацию относительно новой `GameVersion`;
- `ready` может быть сохранен только если новая версия совместима по схеме обязательных слотов и повторная валидация успешно пройдена.

### 11.4 Соревнование
- во время активного соревнования синхронизация игры запрещена;
- состав участников фиксируется по правилам соревнования;
- бан участника или команды во время соревнования должен прервать текущие матчи этой сущности;
- порядок трактовки прерванных матчей задается `ban_resolution_policy`.

### 11.5 Одиночная задача
- пользователь не может иметь более одного активного запуска по всей платформе;
- ограничение активного запуска проверяется по `user_id`, даже если попытка исполняется от имени команды или однослотовой команды;
- запуск новой попытки невозможен, пока предыдущая не завершена или не остановлена;
- история попыток сохраняется;
- лидерборд показывает лучший результат, но хранится вся история.

### 11.6 Worker plane
- `Run` может быть назначен только на `WorkerNode` со статусом `online`;
- worker в состоянии `draining` не получает новые `Run`, но может завершать уже запущенные;
- worker в состоянии `offline` или `disabled` не получает новые `Run`;
- source of truth по статусам запусков остается у control plane, а не у worker.

## 13. Режимы пользовательского API

### 13.1 `turn_based`
Игровой движок вызывает код участника пошагово.

Поддерживаемые варианты entrypoint:
- модульная функция `make_move(observation)`;
- класс `Bot` с методом `make_move`.

Разрешено:
- хранить состояние между ходами в памяти процесса;
- использовать глобальные переменные или состояние экземпляра класса.

Рекомендуемый шаблон:
```python
class Bot:
    def __init__(self):
        self.memory = {}

    def on_match_start(self, context):
        self.memory["map_size"] = context["map_size"]

    def make_move(self, observation):
        return {"action": "wait"}
```

### 13.2 `script_based`
Для задач в стиле CodeCombat участник пишет "сценарный" код через SDK:
- без `input()`;
- с доступом к объекту SDK;
- с возможностью писать `while` и последовательность действий.

Пример:
```python
from sdk import hero

while hero.can_continue():
    if hero.see_coin():
        hero.move_to_coin()
    else:
        hero.turn_right()
```

### 13.3 Управляемое исполнение `script_based`
Платформа обязана исполнять `script_based` код управляемо:
- не давать пользовательскому коду бесконтрольно захватывать процесс навсегда;
- прерывать исполнение после каждого игрового действия SDK;
- продолжать выполнение только под контролем движка;
- вести историю шагов и логов.

Внутренняя реализация может использовать generator/coroutine/cooperative runtime, скрытый от пользователя.

### 13.3.1 Нормативная модель исполнения `script_based`
Для целей реализации и кодогенерации `script_based` трактуется так:
- пользователь пишет обычный Python-код, который выглядит как непрерывный сценарий;
- источником взаимодействия с миром являются только объекты SDK, предоставленные игрой;
- каждый mutating SDK-вызов или явный `sdk.wait()` считается одним игровым действием;
- после каждого игрового действия выполнение пользовательского сценария обязано быть приостановлено;
- при следующем тике движок обязан продолжить выполнение с точной следующей точки пользовательской программы;
- локальные переменные, глобальные переменные и состояние объектов пользователя сохраняются между такими возобновлениями до завершения запуска.

### 13.3.2 Что обязана гарантировать реализация
Реализация `script_based` runtime обязана гарантировать:
- детерминированную точку возобновления после каждого игрового действия;
- ограничение времени вычислений между двумя соседними игровыми действиями;
- невозможность бесконечного CPU-loop без SDK-вызова;
- корректное завершение сценария, если пользовательский код дошел до конца;
- перевод запуска в `timeout` или `failed`, если между двумя игровыми действиями превышен лимит времени или лимит инструкций.

### 13.3.3 Поддерживаемые конструкции Python в `script_based`
Гарантированно поддерживаются:
- переменные;
- условия;
- циклы;
- функции;
- простые классы;
- списки, словари, множества, кортежи;
- арифметика, строки, базовые модули из allowlist runtime profile;
- обработка исключений в пределах одного пользовательского сценария.

### 13.3.4 Что не гарантируется и должно считаться запрещенным
В `script_based` не должны поддерживаться как часть публичного контракта:
- `exec` и `eval`;
- `async`/`await`;
- потоки и процессы;
- monkey patching SDK-объектов;
- интроспекция стека исполнения и code objects;
- динамический импорт модулей, не разрешенных runtime profile;
- любые механики, позволяющие обойти cooperative runtime модели.

### 13.3.5 Формальный lifecycle одного тика `script_based`
Нормативный runtime lifecycle для одного тика:
1. движок возобновляет пользовательский сценарий из последней детерминированной точки;
2. сценарий выполняется до первого терминального события:
   - mutating SDK-вызов;
   - `sdk.wait()`;
   - завершение пользовательской программы;
   - необработанное исключение;
   - превышение лимита инструкций/времени между SDK-действиями;
3. движок фиксирует результат этого тика (action intent / completion / error / timeout);
4. движок приостанавливает сценарий и возвращает управление orchestration-слою.

Обязательное следствие:
- двумя соседними тиками считается участок исполнения между двумя соседними точками cooperative-yield;
- этот участок всегда ограничивается `inter_action_cpu_budget` и `inter_action_wall_time_budget`.

### 13.4 `input()` и `print()`
- `input()` запрещен;
- платформа валидирует пользовательский код при сохранении слота и отклоняет `input()` с `422 invariant_violation`, не дожидаясь запуска;
- `print()` разрешен только как отладочный вывод;
- отладочный вывод виден только автору попытки и teacher/admin;
- отладочные логи не видны зрителям.

### 13.5 Стандартная библиотека Python
Разрешается почти вся стандартная библиотека Python, кроме опасных модулей и API, нарушающих модель песочницы.

Обязательная denylist для `v2`:
- `subprocess`
- `socket`
- `asyncio.subprocess`
- `multiprocessing`
- `ctypes`
- прямые process-control API

Финальный denylist можно расширять настройками runtime profile.

## 15. Git-синхронизация игр

### 15.1 Добавление игры из git
Teacher/admin добавляет игру через UI, указывая:
- URL публичного git-репозитория.

### 15.2 Синхронизация
Синхронизация запускается только вручную.

Алгоритм:
1. скачать или обновить репозиторий;
2. взять `HEAD` default branch;
3. считать `manifest.yaml`;
4. провалидировать структуру пакета;
5. собрать runtime image в изолированном Docker build;
6. собрать renderer bundle;
7. создать новую `GameVersion`;
8. сделать новую версию активной.

### 15.2.1 Безопасность pipeline синхронизации
Синхронизация и сборка игры из git обязаны выполняться в отдельном изолированном builder environment.

Обязательные требования:
- builder не работает в процессе `api-service`;
- builder не имеет доступа к секретам платформы, production env, session storage и базе данных платформы;
- build context ограничен только checkout-копией конкретного репозитория игры;
- в build pipeline нельзя пробрасывать хостовые директории с кодом платформы;
- кэш сборки, если используется, не должен содержать секреты и должен быть отделен от runtime среды платформы;
- сетевой доступ builder ограничивается только необходимыми внешними адресами для clone и разрешенных package registries;
- builder исполняется в отдельном rootless-runtime (или эквивалентной безпривилегированной песочнице), а не в доверенном privileged-контуре платформы;
- build-этап должен стартовать с политикой deny-by-default по сети; доступ в сеть открывается только на allowlist endpoints (git host и разрешенные registries);
- артефакты build (runtime image digest, renderer artifact hash) фиксируются как immutable и проходят policy-check до активации версии;
- если validation, build или security checks не пройдены, текущая активная версия игры не меняется;
- активация новой `GameVersion` разрешена только после полного прохождения validation и build этапов.

### 15.3 Правила обновления активной игры
Если идет активное соревнование по игре:
- синхронизация запрещена.

Если по игре активны тренировочные лобби:
- перед переключением версии выполняется проверка совместимости slot schema;
- совместимым считается обновление, которое не удаляет существующие обязательные слоты, не переименовывает их и не повышает необязательный слот до обязательного для уже существующих лобби;
- добавление новых необязательных слотов допускается;
- если схема слотов несовместима, sync завершается ошибкой `GAME_SYNC_INCOMPATIBLE_SLOT_SCHEMA`, а активная версия не меняется;
- в `v2` режим принудительного переключения на несовместимую slot schema не поддерживается;
- все текущие матчи принудительно останавливаются;
- их статус становится `canceled`, а `error_message` получает reason-code `canceled_by_game_update`;
- результаты не попадают в лидерборд;
- лобби перезапускаются на новой версии игры;
- команды остаются в лобби;
- каждая команда проходит повторную валидацию против новой версии;
- статус `ready` сохраняется только у команд, прошедших повторную валидацию.

### 15.4 Версионирование
Каждая синхронизация создает новую неизменяемую версию игры.

Даже если игра "сразу переключается", commit SHA должен храниться и быть видимым в UI и аудите.

## 16. Команды и коды команды

### 16.1 Команда
Команда создается для конкретной игры.

У команды:
- один капитан;
- набор слотов;
- коды по слотам;
- история снимков кода.

### 16.2 Слоты команды
Слоты и их обязательность задаются игрой.

Примеры:
- `runner`
- `defender`
- `carrier`
- `scout`

### 16.3 Шаблоны кода
Игра сама поставляет шаблоны кода по слотам.

### 16.4 Снимок кода для матча
Перед стартом матча создается неизменяемый snapshot:
- по одному snapshot на слот;
- snapshot привязывается к матчу или попытке;
- дальнейшие изменения кода не влияют на уже стартовавший запуск.

### 16.4.1 Момент фиксации snapshot
Для целей реализации момент фиксации snapshot определяется строго:
- для `Match` это переход `created -> queued`;
- для `SingleTaskAttempt` это переход `created -> queued`.

В рамках одной транзакции система обязана:
- создать недостающие `CodeSnapshot`;
- записать идентификаторы snapshot в `Run.input_payload_json` или эквивалентную неизменяемую структуру входа запуска;
- перевести матч или попытку в `queued`.

После завершения этой транзакции никакие последующие изменения командного кода не могут влиять на конкретный `Match` или `SingleTaskAttempt`.

### 16.5 Изменение кода в очереди
До момента перехода `created -> queued` матч или попытка могут быть пересобраны на основе более нового кода команды.

После перехода в `queued` используется уже зафиксированный snapshot, даже если worker еще не начал фактическое исполнение.

Изменение кода команды само по себе не должно автоматически снимать команду с `ready`.

## 17. Тренировочные лобби

### 17.1 Создание лобби
Лобби всегда создает только `teacher` или `admin`.

### 17.2 Типы доступа
Поддерживаются:
- открытое лобби;
- лобби по коду.

### 17.3 Жизненный цикл
Статусы:
- `draft`
- `open`
- `running`
- `paused`
- `updating`
- `closed`

### 17.4 Ready-state
Вход в лобби не означает готовность.

Команда может быть:
- `not_ready`
- `ready`
- `blocked`

### 17.5 Ограничение готовности
Команда не может стать `ready`, если не заполнены все обязательные слоты.

### 17.6 Пул матчей
Внутри одного лобби может параллельно идти несколько матчей.

### 17.7 Команда в нескольких лобби
Одна команда может участвовать в нескольких лобби одновременно, если у нее есть доступ.

### 17.8 Выход из тренировочного лобби
Если матч еще не идет, команда просто покидает лобби.

Если команда выходит из активного тренировочного `massive_lobby`:
- она исчезает из мира сразу;
- это не требует физического присутствия пользователя онлайн.

## 18. Соревнования

### 18.1 Соревнование - отдельная сущность
Соревнование не является просто тренировочным лобби.

У соревнования есть:
- собственный lifecycle;
- собственный набор entrants;
- собственные настройки;
- собственная логика продвижения.

### 18.2 Форматы
Система должна проектироваться так, чтобы поддерживать:
- `single_elimination`
- `round_robin`
- `swiss`

Для `v2` обязательным является как минимум:
- визуализация elimination-bracket;
- табличное представление для форматов, которые не являются классической сеткой.

### 18.3 Матчи с `N` участниками
Соревнование обязано поддерживать матчи с числом команд больше 2.

### 18.4 Правило продвижения
Правило продвижения настраивается.

Примеры:
- дальше проходит только победитель;
- дальше проходят `top-2`;
- дальше проходят `top-k`.

### 18.4.1 Политика разрешения ничьих
У соревнования обязана быть явно задана `tie_break_policy`.

Поддерживаемые варианты:
- `manual` - соревнование приостанавливается для teacher/admin decision;
- `shared_advancement` - все участники, разделившие граничное место, проходят дальше;
- `tiebreak_match` - автоматически создается дополнительный матч, если игра и формат это позволяют;
- `game_defined` - игра сама возвращает детерминированные `placements` без неоднозначной границы прохода.

### 18.5 Политика обновления кода в соревновании
Настройка соревнования определяет, когда можно менять код:
- `locked_on_registration`
- `locked_on_start`
- `allowed_between_matches`

### 18.6 Бан во время соревнования
Teacher/admin может:
- перевести entrant в `not_ready`;
- забанить entrant.

Если entrant забанен во время активного соревнования:
- все его текущие матчи прерываются;
- трактовка результатов определяется `ban_resolution_policy`.

### 18.7 Проверка схожести решений
Проверка схожести работает только в рамках соревнования.

Она запускается автоматически при каждом новом используемом snapshot в соревновании.

Автоматического наказания нет.

Teacher/admin получает warning.

## 19. Матчи и попытки

### 19.1 Общий lifecycle матча
Статусы:
- `created`
- `queued`
- `running`
- `finished`
- `failed`
- `canceled`
- `aborted`

### 19.2 Lifecycle одиночной попытки
Статусы:
- `created`
- `queued`
- `running`
- `finished`
- `stopped`
- `failed`
- `timeout`
- `canceled`

### 19.2.1 Причины отмены run (`error_message`)
Для статуса `canceled` API возвращает машинно-читаемый `error_message` с кодом причины.

В `v2` фиксируется следующий словарь кодов:
- `manual_cancel` - ручная остановка через `POST /runs/{runId}/cancel`;
- `manual_stop_single_task` - ручная остановка single-task попытки через `POST /single-tasks/{gameId}/stop`;
- `manual_moderation_ban` - отмена из-за модерационного бана участника в активном соревновании;
- `canceled_by_game_update` - отмена из-за обновления версии игры в training-лобби.

Правила:
- backend может добавлять дополнительные коды без breaking change;
- неизвестные коды должны отображаться в UI как "служебная причина" с показом raw-значения;
- `error_message` не используется как человеко-читаемый текст ошибки для локализации, а только как стабильный reason-code.

### 19.3 Результат игры
Игра возвращает result payload.

Минимальный контракт:
```json
{
  "status": "finished",
  "winner_ids": ["team-1"],
  "scores": {
    "team-1": 15,
    "team-2": 8
  },
  "placements": {
    "team-1": 1,
    "team-2": 2
  },
  "team_stats": {},
  "player_stats": {},
  "replay_ref": "replay-uuid",
  "metrics": {
    "duration_ms": 25000,
    "ticks": 420
  }
}
```

Правила обязательности полей:
- для `single_task` обязательны `status`, `metrics`, `replay_ref` при наличии реплея;
- для тренировочных multiplayer-матчей обязательны `status` и хотя бы один из блоков `scores` или `placements`;
- для соревновательных матчей обязательны `status`, `placements` и `metrics`;
- если игра возвращает только `scores` для соревновательного матча и не возвращает `placements`, такой результат считается невалидным и матч не может быть использован для турнирного продвижения.

### 19.3.1 Политика равенства результатов
Если у нескольких участников одинаковый `score`, игра обязана:
- либо явно вернуть одинаковые `placements` и тем самым зафиксировать ничью;
- либо вернуть уже разрешенные детерминированные `placements`.

Платформа не должна самостоятельно выводить турнирные места из `scores`, если `placements` отсутствуют.

### 19.4 Что является источником истины для результата
Источником истины является payload, возвращенный игрой.

Платформа не должна вычислять победителя за игру, если игра сама уже вернула placements и scores.

Для соревнований источником истины для продвижения являются именно `placements` с учетом `tie_break_policy`.

## 20. Single-player задачи

### 20.1 Метаданные задачи
Для `single_task` обязательны:
- описание;
- сложность;
- темы.

Для `v2` источником истины для этих полей является UI/БД платформы, а не manifest репозитория.

### 20.1.1 Onboarding single-task игры после sync
Если новая `single_task` игра синхронизирована из репозитория:
- платформа создает техническую запись игры и версии;
- README и `player_guide_ru.md`, если существуют, используются как исходный материал для первичного заполнения формы;
- игра получает `catalog_metadata_status = draft`, пока teacher/admin не заполнит обязательные поля каталога в UI;
- до перевода `catalog_metadata_status` в `ready` такая игра не должна отображаться в публичном каталоге задач.

Текущая реализация MVP:
- `catalog_metadata_status` возвращается в `GameResponse`;
- для `single_task` регистрация без `description/difficulty/topics` создает игру в `draft`;
- endpoint `GET /catalog/single-tasks` возвращает только `single_task` игры со статусом `ready`.

### 20.1.2 Precedence policy для каталожных полей
При конфликте между репозиторием и UI/БД:
- описание задачи, сложность и темы берутся из UI/БД;
- README, SDK guide, шаблоны и технические параметры берутся из игрового пакета;
- UI/БД может override только каталожные поля, а не технический контракт игры.

### 20.2 Темы
Одна задача может относиться к нескольким темам.

Примеры:
- графы;
- BFS/DFS;
- жадные алгоритмы;
- динамическое программирование;
- симуляция.

### 20.3 Сложность
Сложность хранится как минимум в одном из значений:
- `easy`
- `medium`
- `hard`

### 20.4 Лидерборд
Если у задачи включен leaderboard:
- хранится история всех попыток;
- в таблицу попадает лучший результат пользователя.

Если у задачи нет score-модели:
- можно показывать список решивших пользователей.

### 20.5 Глобальная метрика решенных задач
Платформа должна уметь показывать, кто решил больше задач.

## 21. Spectator mode и реплеи

### 21.1 Два режима просмотра
Поддерживаются:
- live-view;
- replay после завершения.

### 21.2 Публичность
Если:
- игра поддерживает spectator mode;
- и конкретное лобби/соревнование разрешает публичный доступ,

то матч и реплей доступны без логина.

### 21.3 Отображение на уровне лобби
Страница лобби или соревнования показывает список матчей.

Зритель выбирает конкретный матч и открывает его.

### 21.4 Безопасность renderer
Renderer из стороннего репозитория:
- работает в `iframe`;
- не получает auth tokens пользователя;
- не обращается к приватным API напрямую.

Оболочка SPA:
- получает данные сама;
- передает renderer'у только разрешенные события и состояния через `postMessage`.

### 21.5 Что передает shell renderer'у
- начальную конфигурацию игры;
- поток кадров;
- события матча;
- итоговый result payload;
- данные реплея.

### 21.6 Реплеи
Реплей должен хранить:
- кадры;
- события;
- финальный summary;
- версию игры;
- commit SHA игры;
- конфигурацию запуска.

Эти данные могут храниться либо прямо в записи `Replay`, либо быть детерминированно восстанавливаемыми через связку `Replay -> Match -> GameVersion`, но API реплея обязан отдавать их как часть единого ответа без дополнительных догадок клиента.

## 22. Авторизация GeekClass

Для `v2` авторизация переносится 1-в-1 по аналогии с `GeekCodeBattle`.

### 22.1 Login flow
1. backend делает redirect на `{GEEKCLASS_HOST}/insider/jwt?redirect_url={callback_url}`;
2. GeekClass возвращает JWT в query-параметре `token`;
3. backend декодирует JWT по `JWT_SECRET`, `HS256`;
4. backend проверяет `iat`, TTL токена - 60 секунд;
5. backend делает upsert локального пользователя;
6. backend сохраняет вход в server-side session.

### 22.1.1 Session model
Внутренний auth-state платформы после успешного входа обязан храниться как server-side session:
- браузер получает session cookie;
- cookie должна быть `HttpOnly`;
- cookie должна иметь безопасные флаги для production (`Secure`, подходящий `SameSite`);
- SPA не хранит собственный long-lived access token для основной работы приложения;
- backend проверяет сессию на каждом защищенном запросе API и WebSocket handshake.

### 22.2 JWT payload
Ожидаемый формат:
```json
{
  "id": 123,
  "role": "student",
  "name": "Иван Иванов",
  "iat": 1713272000
}
```

### 22.3 Dev login
Для локальной разработки обязателен `dev-login`.

Dev login должен позволять:
- войти под любой ролью;
- задать произвольный nickname;
- задать внешний id.

## 23. Песочница и исполнение

### 23.1 Обязательные ограничения Docker
- без `--privileged`;
- без доступа к Docker socket;
- без доступа к хостовой БД;
- изолированная файловая система;
- временная рабочая директория;
- автоматическое удаление контейнера после завершения;
- лимиты CPU, RAM, PIDs;
- лимит времени;
- сеть настраивается на уровне игры.

### 23.2 Коды участников и игровые движки
Исполняются изолированно:
- код участника - в изолированном runtime;
- игровой движок - в отдельном Docker-контейнере игры;
- платформа не исполняет game code в своем собственном процессе.

### 23.3 Worker model
Компоненты:
- `api-service`
- `scheduler-service`
- `builder-service`
- `game-worker`
- `redis`
- `postgres`

Архитектурно система делится на:
- `control plane`: `api-service`, `scheduler-service`, `postgres`, `redis`
- `build plane`: `builder-service`
- `execution plane`: множество `game-worker`, которые могут располагаться как локально, так и на удаленных серверах.

### 23.4 Планировщик
Планировщик:
- выбирает свободный worker;
- ставит job в очередь;
- отслеживает heartbeat;
- обновляет статус запуска;
- обрабатывает таймауты.

Дополнительно планировщик обязан:
- быть единственным сервисом, который принимает решение о назначении `Run` на конкретный worker;
- использовать как источник истины `PostgreSQL` для жизненного цикла `Run`;
- использовать `Redis` как coordination layer, а не как долговременный source of truth;
- уметь работать с worker, находящимися на удаленных хостах.

### 23.5 Heartbeat
Каждый worker периодически сообщает:
- свою доступность;
- число свободных слотов;
- активные запуски;
- текущую нагрузку.

Минимальный heartbeat payload должен включать:
- `worker_id`
- `hostname`
- `available_slots`
- `running_run_ids`
- `supported_runtime_profiles`
- `last_seen_at`
- агрегированную информацию по CPU/RAM

### 23.6 `builder-service`
`builder-service` отвечает только за build/sync pipeline игровых пакетов:
- clone/update git-репозитория игры;
- validation структуры пакета;
- build runtime image;
- build renderer artifact;
- публикацию или подготовку immutable артефактов для исполнения.

`builder-service` не должен:
- исполнять пользовательские матчи;
- участвовать в realtime матчей;
- совмещаться с `api-service` в одном процессе.

### 23.7 `game-worker`
`game-worker` отвечает только за исполнение уже подготовленных запусков.

`game-worker` обязан:
- получать задачу исполнения после назначения от `scheduler-service`;
- запускать только готовые runtime image по immutable image reference / digest;
- не выполнять build из git-репозитория во время матчевого исполнения;
- регулярно отправлять heartbeat;
- публиковать lifecycle events и игровые события обратно в control plane;
- корректно завершать и очищать контейнеры после завершения `Run`.

### 23.8 Remote workers
`game-worker` может размещаться:
- на том же хосте, что и control plane;
- на отдельном сервере в той же сети;
- на нескольких удаленных серверах для горизонтального масштабирования.

С точки зрения платформы новый сервер масштабирования - это просто еще один зарегистрированный `game-worker`.

### 23.8.1 Worker labels и операционный профиль
Для удаленных worker-нод используется конфигурируемый набор меток (`worker labels`), например:
- `region`;
- `pool`;
- `hardware_class`.

`worker labels`:
- регистрируются вместе с worker в control plane;
- используются как metadata для observability и операционного управления;
- участвуют в scheduler-распределении через `required_worker_labels` у `Run` (exact-match по заданным ключам).

Операционный runbook и пример запуска удаленных worker описаны в `docs/REMOTE_WORKER_SCALING_V1.md`.

### 23.9 Принцип доступа worker к данным платформы
Для `v2` предпочтительная модель такая:
- source of truth по `Run`, `Match`, `Attempt` и их статусам хранится в `PostgreSQL`;
- `scheduler-service` обновляет эти статусы как управляющий компонент;
- `game-worker` не обязан иметь прямой доступ к `PostgreSQL` и не должен быть зависим от секретов основной БД, если этого можно избежать;
- обмен управляющими сообщениями и техническими событиями между control plane и worker должен идти через `Redis` и/или внутренние сервисные endpoints.

### 23.10 Последовательность распределенного запуска
Нормативная схема запуска:
1. `api-service` создает доменную сущность `Match` или `SingleTaskAttempt`.
2. `scheduler-service` переводит запуск в `queued`, фиксирует snapshot и создает/обновляет `Run`.
3. `scheduler-service` ставит `Run` в очередь (`/internal/runs/{runId}/schedule`).
4. `game-worker` запрашивает задачу (`/internal/workers/pull-next`) и получает lease на `Run`.
5. `game-worker` фиксирует прием запуска в control plane (`/internal/runs/{runId}/accepted`) и после этого переводит `Run` в `running` (`/internal/runs/{runId}/started`).
6. `game-worker` запускает Docker runtime.
7. `game-worker` публикует heartbeat, lifecycle events, `FRAME`, `EVENT`, `STATS`, `MATCH_RESULT`.
8. после завершения worker подтверждает обработку (`/internal/runs/ack-finished`), а control plane закрывает `Run`.

### 23.10.1 Lease TTL и повторная постановка в очередь
Для устойчивости распределенного запуска вводится обязательная lease-модель:
- при выдаче `Run` через `/internal/workers/pull-next` scheduler создает lease с `leased_at` и `worker_id`;
- каждый lease имеет `lease_ttl_seconds` (конфигурируемый параметр scheduler);
- если lease истек, а `Run` все еще активен (`known`), scheduler обязан автоматически снять истекший lease и вернуть `Run` обратно в очередь;
- повторная постановка должна быть дедуплицирована, чтобы один и тот же `Run` не появлялся в очереди несколько раз одновременно;
- подтверждение `/internal/runs/ack-finished` считается валидным только от того worker, у которого актуальный lease;
- `ack` от устаревшего или неактуального lease (`stale ack`) не должен закрывать `Run` и не должен удалять его из активного множества.

### 23.11 Политика отказа worker
Если worker перестал отправлять heartbeat:
- `scheduler-service` помечает worker как unavailable;
- новые `Run` на него не назначаются;
- зависшие запуски помечаются как `failed` или переводятся в requeue по отдельной retry policy;
- source of truth о результате все равно остается в control plane.

### 23.11.1 Retry/backoff для внутренних HTTP вызовов worker
`game-worker` обязан использовать retry/backoff для временных ошибок внутренних API:
- retry применяется к transport-ошибкам и HTTP `5xx`/`408`/`425`/`429`;
- retry не применяется к детерминированным клиентским ошибкам (`4xx`, кроме перечисленных выше);
- стратегия backoff должна быть экспоненциальной с ограничением максимальной задержки;
- параметры retry (`max_attempts`, `base_delay`, `max_delay`) должны быть конфигурируемыми.

## 25. Матчмейкинг

### 25.1 Для тренировочных лобби
Матчмейкинг - настраиваемое свойство лобби:
- `random`;
- `skill_based`.

Если выбран `skill_based`, платформа использует внутреннюю скрытую метрику силы команды для конкретной игры.

Для `v2` эта метрика определяется как простая эвристика на основе недавних результатов команды:
- placements;
- scores;
- доля побед или успешных попыток;
- экспоненциальное затухание старых результатов.

Если данных недостаточно, `skill_based` обязан деградировать к `random`.

### 25.2 Для соревнований
Посев - настраиваемое свойство соревнования:
- `random`;
- `rating`;
- `manual`.

### 25.3 Командный матчмейкинг
Матчмейкинг работает по командам, а не по отдельным пользователям.

- При наличии хотя бы одного source в `syncing` экран включает авто-поллинг списка/истории.
- Для выбранного source доступна панель диагностики: `requested_by`, `build_id`, `commit_sha`, `image_digest`, `error_message`.
- Teacher/admin может вручную переключать source между `active` и `disabled`; для `disabled` запуск sync блокируется policy backend.
- На том же экране есть секция `Worker Control` для переключения статуса worker (`online|offline|draining|disabled`) и просмотра емкости.

## 30. Антиплагиат

### 30.1 Область действия
Сравнение выполняется только в рамках одного соревнования.

### 30.2 Запуск
Запускается автоматически при каждом новом snapshot, используемом в соревновании.

### 30.3 MVP-алгоритм
Для `v2` достаточно локальной эвристики:
- нормализация кода;
- token similarity;
- AST similarity;
- строковое сравнение;
- итоговый weighted score.

### 30.4 Реакция системы
Система:
- не наказывает автоматически;
- создает warning teacher/admin;
- показывает пары похожих решений;
- показывает score сходства;
- сохраняет audit trail review.

## 31. Обновление игры и влияние на активные объекты

### 31.1 Если идет соревнование
Синхронизация игры запрещена.

### 31.2 Если идет тренировочный матч
Матч останавливается и помечается как `canceled`, reason-code: `canceled_by_game_update`.

### 31.3 Если есть открытые лобби
Лобби перезапускаются:
- остаются на месте;
- переключаются на новую `GameVersion`;
- сохраняют команды;
- выполняют повторную валидацию команд против новой версии;
- сохраняют `ready` только у команд, которые остаются совместимыми с новой схемой обязательных слотов;
- переводят остальные команды в `not_ready` с понятной причиной в UI.

## 36. Критерии приемки

1. Пользователь может войти через GeekClass JWT и через dev-login в development.
2. Teacher/admin может зарегистрировать игру из публичного git-репозитория и вручную синхронизировать ее.
3. После синхронизации создается новая `GameVersion` с commit SHA и активируется.
4. Teacher/admin может создать тренировочное лобби для встроенной или git-игры.
5. Команда может войти в лобби даже без полного набора кодов, но не может стать `ready`, пока обязательные слоты пусты.
6. Одно лобби может запускать несколько матчей параллельно.
7. Одна команда может одновременно находиться в нескольких лобби.
8. При обновлении игры активные тренировочные матчи останавливаются, а лобби перезапускаются на новой версии с повторной валидацией `ready`-состояния команд.
9. Во время активного соревнования обновление игры запрещено.
10. `single_task` не позволяет одному пользователю иметь два активных запуска одновременно по всей платформе.
11. Платформа в целом поддерживает и `turn_based`, и `script_based` API, а конкретная версия игры явно объявляет один режим `code_api_mode`.
12. `script_based` режим работает без `input()` и под контролем движка по шагам.
13. Renderer сторонней игры работает в `iframe`, не имея прямого доступа к auth tokens.
14. Публичный spectator mode позволяет смотреть live и replay без логина, если игра и лобби это разрешают.
15. Проверка схожести решений автоматически создает warning teacher/admin в рамках соревнования.
16. Есть тесты для доменной логики, критичных API и worker orchestration.

## 38. Инструкция для будущей кодогенерации

При генерации кода обязательно соблюдать:
- DDD-структуру и bounded contexts из этого документа;
- явные доменные инварианты;
- тесты на каждую нетривиальную бизнес-операцию;
- отсутствие бизнес-логики в контроллерах;
- транзакционность критичных операций;
- идемпотентность worker-команд и game events;
- при адаптации legacy-игр: сначала тест, воспроизводящий проблему, затем исправление;
- использование `SVG` как формата по умолчанию для новой графики (игровые ассеты, иконки, схемы), если нет явной причины для raster;
- хорошую читаемость и качество кода вместо "быстрого, но хрупкого" решения.

Если какой-либо участок можно сделать проще без потери требований, выбирать более простую реализацию.

Если какой-либо участок нельзя реализовать без дополнительного решения, по умолчанию использовать:
- безопасное поведение;
- детерминированное поведение;
- минимально сложное поведение.
