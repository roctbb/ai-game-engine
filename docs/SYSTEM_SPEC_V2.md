## 2. Цель продукта
Платформа предназначена для:
- обучения программированию через интерактивные визуализируемые задачи;
- проведения тренировочных матчей и соревнований между ботами;
- поддержки командных и индивидуальных сценариев;
- запуска пользовательского кода в изолированной инфраструктуре;
- подключения новых игр из репозитория по стандартному протоколу.

## 3. Основные продуктовые режимы

### 3.1 `single_task`
Одиночная задача, в которой участвует один пользователь или одна команда из одного слота.

Примеры:
- выход из лабиринта;
- сбор монет при движении по карте вправо и вниз;
- расстановка башен для защиты базы;
- задачи в стиле CodeCombat через игровой SDK.

Свойства:
- безлимитное число попыток;
- запрещен параллельный запуск двух решений одним и тем же пользователем по всей платформе;
- следующий запуск возможен только после завершения или явной остановки предыдущего;
- результаты, история попыток, реплеи и отладочные логи сохраняются;
- лидерборд может быть включен или выключен для конкретной игры;
- если у игры нет score-модели, платформа может показывать только список успешно решивших.

### 3.2 `multiplayer`
Матчи с настраиваемым числом участников или команд.

Примеры:
- крестики-нолики до 5 на большой доске;
- дуэльные игры;
- игры 3v3 или free-for-all на ограниченное число участников.

Свойства:
- есть тренировочные лобби;
- есть соревнования;
- игра задает `min_players_per_match` и `max_players_per_match`;
- старые продуктовые различия "маленькое лобби" и "большое лобби" больше не являются отдельными режимами;
- `small_match` и `massive_lobby` допустимы только как legacy aliases при миграции старых игр и должны нормализоваться в `multiplayer`;
- матч стартует, когда готовых свободных игроков не меньше `min_players_per_match`;
- если готовых игроков больше `max_players_per_match`, матчмейкинг запускает несколько параллельных матчей по тем же правилам;
- участники, которых достаточно для еще одного матча, не ждут окончания предыдущего матча, а попадают в следующий параллельный матч того же батча;
- участники, которых не хватает до `min_players_per_match`, остаются в очереди до следующего цикла;
- продвижение по турниру настраивается;
- игра может возвращать очки, а турнир использует правила интерпретации результатов.

### 3.3 Пользовательские сценарии

Этот раздел фиксирует сценарии использования как основной источник будущей структуры frontend.

Цель раздела:
- отделить реальные пользовательские намерения от текущего набора страниц;
- убрать навигационную путаницу между каталогом, workspace, run page, лобби, матчами и replay;
- задать приоритетные flows, вокруг которых должен быть собран новый UI;
- сделать явным, какие действия являются основными для разных ролей.

Frontend не должен строиться как набор равноправных CRUD-страниц. Он должен вести пользователя по сценариям: выбрать задачу, написать код, запустить, понять результат, улучшить решение, подготовить команду, войти в матч, посмотреть replay, провести соревнование или обслужить платформу.

#### 3.4.1 Роли в сценариях

Основные роли:
- `student` - решает задачи, пишет код команды, участвует в лобби и соревнованиях, смотрит свои результаты;
- `teacher` - организует лобби и соревнования, подключает игры, модерирует участников, разбирает результаты;
- `admin` - управляет платформенными настройками, game sources, runtime profiles и worker plane.

UI обязан всегда показывать пользователю только те первичные действия, которые доступны его роли и текущему состоянию объекта.

#### 3.4.2 Сценарий A: студент решает одиночную задачу

Цель пользователя:
- выбрать задачу;
- понять условие;
- написать или изменить решение;
- запустить попытку;
- увидеть визуализацию, результат, логи и replay;
- понять, как его решение работает пошагово;
- сравнить свой результат с другими пользователями, которые решали эту задачу;
- улучшить решение и повторить цикл.

Основной flow:
1. Пользователь открывает каталог задач.
2. Видит задачи сгруппированными по темам и сложности.
3. Фильтрует или сворачивает группы по теме, сложности или статусу решения.
4. Открывает задачу.
5. Видит условие, документацию задачи/игры, текущий прогресс, шаблон кода и canonical workspace для этой задачи.
6. Редактирует код.
7. Запускает попытку.
8. Во время выполнения видит viewer, статус run и debug output.
9. После завершения видит result summary, replay, метрики и историю попыток.
10. Видит свое место или позицию относительно других решений: leaderboard, percentile, число решивших, лучший score/время/метрики, если эти данные доступны.
11. Возвращается к редактированию кода без потери контекста.

Обязательные UX-правила:
- `single_task` run page не является отдельным хранилищем кода;
- редактор на run page редактирует тот же `TeamSlotCode`, что и workspace;
- документация игры/задачи является частью задачи и должна быть доступна в контексте task/run page, а не как внешняя справочная страница без связи с задачей;
- каталог задач должен группировать задачи по теме и сложности, а не показывать только плоский список карточек;
- если задача относится к нескольким темам, UI должен иметь стабильную политику отображения: основная тема, повтор в нескольких группах или группировка через topic sections с дедупликацией;
- главный CTA в этом сценарии - `Запустить`;
- визуальный результат и replay являются не декоративным блоком, а основным способом понять, как работал код;
- сравнение с другими пользователями является обязательной мотивационной частью режима задач, если у задачи есть leaderboard или success stats;
- если score-модели нет, UI должен показывать доступное сравнение: решено/не решено, число решивших, последние успешные решения или другое честное агрегированное состояние;
- если у пользователя уже есть активный запуск, UI объясняет блокировку и дает перейти к активному run;
- результат должен объяснять не только `finished/failed/timeout`, но и что можно сделать дальше.

#### 3.4.3 Сценарий B: студент участвует в тренировочном лобби

На текущем этапе подготовка кода и участие в тренировочном лобби являются одним пользовательским сценарием.

У пользователя есть только один игрок для одной игры. В текущем UX игрок и команда фактически означают одно и то же.

Внутри backend может сохраняться техническая сущность `Team`, потому что она уже является контейнером слотов, snapshot и участия в матчах. Но в пользовательском интерфейсе этот объект нужно показывать как игрока/участника лобби, а не как отдельную команду, которую надо выбирать или администрировать.

Игрок в пользовательском смысле - это набор кодов для конкретной игры:
- для простой игры это может быть один код одного игрового агента;
- для командной/ролевой игры это несколько кодов для разных ролей;
- набор ролей и обязательность слотов задаются игрой.

Цель пользователя:
- выбрать подходящее лобби;
- заполнить код для всех обязательных ролей;
- нажать `Готов` / `Play`, когда код заполнен;
- видеть, кто играет, кто готовится и кто стоит в очереди;
- понимать, с кем он играет и насколько хорошо эти игроки выступают;
- попасть в текущую игру;
- смотреть пошаговую визуализацию и статистику;
- при необходимости нажать `Не готов` / `Stop`, выйти из игры или очереди и вернуться к редактированию;
- менять код во время ожидания или игры так, чтобы изменения применялись со следующей игры.

Основной flow:
1. Пользователь открывает список лобби.
2. Фильтрует лобби по игре.
3. Открывает лобби.
4. Если пользователь ранее в это лобби не заходил, платформа создает запись участия в лобби и обеспечивает наличие команды пользователя для игры этого лобби.
   Если команды для этой игры еще нет, она создается автоматически.
5. Пользователь попадает на страницу лобби с тремя вкладками сверху: `Код`, `Лобби`, `Игра`.
6. На вкладке `Код` пользователь видит ссылку на документацию игры и заполняет код для всех обязательных ролей.
7. Как только все обязательные роли заполнены, появляется действие `Готов` / `Play`.
8. После нажатия `Готов` пользователь становится готовым к игре.
   Для `multiplayer` он попадает в очередь до подбора матча по диапазону участников игры.
   Если готовых игроков достаточно для нескольких матчей, они запускаются параллельно одним батчем.
   Следующий батч не стартует, пока текущий батч не завершил симуляцию, replay-период и короткий result-период.
9. На вкладке `Лобби` пользователь видит участников по группам:
   - `играют`;
   - `готовятся`;
   - `в очереди`.
10. На вкладке `Лобби` пользователь видит список текущих матчей и архив матчей этого лобби.
    Матчи, запущенные одновременно, визуально группируются по времени запуска.
11. Просмотр матча из текущих матчей или архива открывается в отдельной вкладке/странице match viewer.
12. По каждому игроку лобби UI показывает доступную статистику: число побед, среднее число очков, недавнюю форму или другие агрегаты, если они доступны.
13. Когда для пользователя находится игра, активной становится вкладка `Игра`.
14. На вкладке `Игра` показывается игра текущего батча, в которой участвует пользователь.
    Если у пользователя есть несколько текущих replay-кандидатов, UI дает компактный переключатель между ними.
15. Вкладка `Игра` использует двухколоночный layout: слева максимально крупная пошаговая визуализация игры, справа статистика текущего матча.
16. Если пользователь нажимает `Не готов` / `Stop`, он выходит из текущей игры или очереди и переходит в статус `готовится`.
17. Если пользователь меняет код и нажимает `Готов` / `Play`, новые изменения применяются со следующей игры.

Обязательные UX-правила:
- lifecycle лобби (`open|paused|updating|closed`) должен быть виден всегда;
- актуальный lifecycle тренировочного лобби: `draft|open|running|paused|stopped|updating|closed`;
- `paused` означает временную паузу без очистки участия;
- `stopped` означает ручную остановку teacher/admin: очередь очищена, участники переведены в подготовку, новые матчи не стартуют до `open`;
- `closed` является terminal-status для старых сценариев закрытия; для полного удаления используется отдельное действие `DELETE`;
- вкладки `Код`, `Лобби`, `Игра` являются главным student UI для тренировочного лобби;
- вкладка `Код` активна, когда нужно заполнить роли или изменить решение;
- вкладка `Код` обязана содержать заметную ссылку на документацию игры как часть текущей игры/задачи;
- если игровой пакет поставляет `docs/player_guide_ru.md` и `docs/sdk_guide_ru.md`, UI должен дать доступ как минимум к player guide и SDK guide;
- вкладка `Лобби` активна, когда пользователь готовится, стоит в очереди или хочет понять состояние лобби;
- вкладка `Лобби` должна помогать игроку понять, с кем он играет и насколько сильны/успешны другие участники;
- вкладка `Игра` активна, когда пользователь участвует в текущей игре;
- действие `Готов` / `Play` доступно только после заполнения всех обязательных ролей;
- для `multiplayer` действие `Готов` / `Play` переводит игрока в очередь до подбора матча по диапазону `min_players_per_match..max_players_per_match`;
- если игрок уже в очереди или текущем батче, UI показывает `Stop`, а не дополнительный `Play`;
- действие `Не готов` / `Stop` переводит пользователя в статус `готовится` и выводит его из очереди или текущей игры;
- статус `готовится` означает, что пользователь находится в лобби, но сейчас не ожидает матч и не участвует в игре;
- список текущих матчей и архив матчей должны быть частью вкладки `Лобби`;
- match viewer для текущего или архивного матча открывается в отдельной вкладке/странице, сохраняя контекст возврата в лобби;
- архив матчей должен показывать дату/время запуска, участников, победителя и постраничное отображение;
- пользователь не должен терять возможность редактировать canonical код во время ожидания или текущей игры;
- UI обязан явно показывать snapshot boundary: текущая игра использует код, зафиксированный при постановке в очередь, а новые сохранения пойдут в следующий запуск.

Цикл тренировочного лобби:
1. `waiting_players`: система ждет минимум `min_players_per_match` готовых свободных игроков.
2. `waiting_viewer`: если готовые игроки есть, но ни один готовый участник не онлайн и не может увидеть матч, система ждет хотя бы одного viewer.
3. `simulation`: все возможные матчи текущего батча запускаются параллельно. Размер каждого матча находится в диапазоне `min_players_per_match..max_players_per_match`.
4. `replay`: после окончания всех run батча пользователи синхронно смотрят replay. Длительность replay-периода считается по самому длинному матчу батча. Базовая скорость replay в лобби - не быстрее 0,5 секунды на кадр.
5. `result`: после replay показывается короткий итоговый период с победителем матча пользователя.
6. После `result` цикл начинается снова, если лобби открыто и есть готовые игроки.

Правила синхронизации replay:
- backend хранит время начала replay-периода батча;
- frontend при входе во вкладку `Игра` вычисляет текущий кадр от `replay_started_at` и `cycle_frame_ms`;
- пользователь, пришедший через несколько секунд после старта replay, сразу видит соответствующий кадр, а не начало;
- если матч уже дошел до конца, кнопка Play в viewer должна начинать replay с начала.

#### 3.4.4 Сценарий C: студент участвует в соревновании внутри лобби

Цель пользователя:
- понять, что в текущем лобби идет соревнование;
- следить за турнирной сеткой в реальном времени;
- видеть свой текущий матч во вкладке `Игра`;
- открывать матчи соревнования для просмотра;
- видеть, когда код заблокирован политикой соревнования.

Основной flow:
1. Пользователь находится в лобби.
2. Teacher/admin запускает соревнование из этого лобби.
3. Лобби показывает заметную плашку `Идет соревнование`.
4. Лобби блокирует вход новых участников и выход текущих участников.
5. На странице лобби появляется вкладка `Соревнование`.
6. Во вкладке `Соревнование` пользователь видит обновляемую в реальном времени турнирную сетку.
7. В сетке видно, кто сейчас играет, а после завершения матча - победитель и проигравшие.
   Матчи одного этапа соревнования запускаются параллельно, когда для этого этапа известны все участники.
8. Клик по матчу открывает match viewer для просмотра live или replay.
9. Если пользователь участвует в текущем матче, его игра показывается во вкладке `Игра`.
10. После завершения всех матчей и выбора победителя соревнование остается доступным во вкладке `Соревнование`, пока teacher/admin не нажмет `Завершить соревнование`.
11. После завершения teacher/admin вкладка `Соревнование` пропадает.
12. Если в лобби было хотя бы одно соревнование, появляется вкладка `Архив соревнований`, где доступны итоги прошлых соревнований.

Обязательные UX-правила:
- соревнование не должно уводить student-flow на отдельную самостоятельную страницу, если оно прикреплено к лобби;
- вкладка `Соревнование` видна только во время активного или ожидающего ручного завершения соревнования;
- вкладка `Архив соревнований` видна только если в этом лобби уже было хотя бы одно завершенное соревнование;
- лобби во время соревнования должно визуально отличаться от обычной тренировки через persistent banner;
- действия входа и выхода из лобби во время соревнования заблокированы;
- bracket не должен быть единственным представлением, если формат плохо укладывается в бинарную сетку;
- для матчей с `N > 2` участниками UI обязан явно показывать всех участников и правило продвижения;
- этапы соревнования должны использовать тот же принцип батчей, что и training lobby: сначала параллельно запускаются все матчи этапа, затем показываются replay/result и только после этого формируется следующий этап;
- блокировка редактирования кода должна быть объяснена через `code_policy`;
- спорные tie-break состояния должны быть видны как отдельная остановка сценария.

#### 3.4.5 Сценарий D: зритель смотрит live match или replay

Цель пользователя:
- быстро понять, что происходит в матче;
- видеть крупную визуализацию;
- иметь доступ к счету, событиям, timeline и итогам;
- смотреть публичные матчи без логина только после появления явного public spectator mode.

Основной flow:
1. Пользователь открывает матч из лобби, соревнования или каталога replay.
2. SPA загружает watch-context и renderer в iframe.
3. Shell показывает состояние загрузки renderer.
4. Renderer сообщает `agp.renderer.ready`.
5. Shell передает `init`, затем поток `state`.
6. В terminal state shell передает `result` и показывает summary.
7. Пользователь переключается между live/replay, двигает timeline, смотрит события и метрики.

Обязательные UX-правила:
- viewer является главным объектом страницы;
- renderer не должен ломать layout shell;
- ошибки renderer должны быть видимы и recoverable;
- публичный spectator mode не должен выглядеть как урезанный служебный режим;
- debug logs пользовательского кода не показываются публичным зрителям.

#### 3.4.6 Сценарий E: teacher создает и ведет тренировочное лобби

Цель пользователя:
- создать лобби для игры;
- настроить доступ, лимит игроков, срок автоудаления старых тренировочных матчей, spectator mode и параметры matchmaking;
- управлять lifecycle;
- разбирать проблемные run.

Основной flow:
1. Teacher открывает раздел лобби.
2. Создает лобби для конкретной игры и активной версии.
3. Настраивает `access_mode`, `max_teams`, spectator mode и config.
4. Открывает лобби для участников.
5. Наблюдает за очередью, техническим ready-state команд и активными матчами.
6. При необходимости запускает лобби, ставит на паузу, останавливает, редактирует настройки или полностью удаляет лобби вместе с тренировочными матчами.
7. Может нажать `Начать соревнование`, выбрать настройки соревнования и запустить его внутри этого лобби.
8. Во время соревнования видит плашку `Идет соревнование`, вкладку `Соревнование` и заблокированное состояние входа/выхода.
9. После завершения всех игр и выбора победителя нажимает `Завершить соревнование`, после чего активная вкладка `Соревнование` пропадает, а результаты остаются в `Архиве соревнований`.
10. При обновлении игры видит состояние `updating` и результат повторной валидации команд.
11. Разбирает `canceled/failed/timeout` через run status, reason-code, stream и replay.

Обязательные UX-правила:
- teacher-действия должны быть отделены от student-действий визуально и семантически;
- destructive/lifecycle действия требуют явного контекста объекта;
- полное удаление лобби должно быть отделено от `Stop`: `Stop` сохраняет лобби и историю, `Delete` удаляет лобби и связанные тренировочные матчи;
- teacher/admin может редактировать название, доступ, код входа, лимит игроков и `auto_delete_training_runs_days`;
- `Начать соревнование` доступно только teacher/admin из контекста конкретного лобби;
- настройки соревнования выбираются перед запуском и фиксируются для текущего соревнования;
- во время соревнования teacher/admin не должен случайно открыть лобби для новых участников;
- причина отмены run должна показываться как stable reason-code и человеко-читаемый label;
- `updating` не должен выглядеть как обычная загрузка страницы.

#### 3.4.7 Сценарий F: teacher проводит соревнование

Цель пользователя:
- начать соревнование из лобби;
- настроить формат и правила продвижения;
- зафиксировать состав;
- стартовать, приостанавливать и завершать соревнование;
- разрешать tie-break;
- модерировать участников.

Основной flow:
1. Teacher/admin находится в лобби.
2. Нажимает `Начать соревнование`.
3. Выбирает настройки: формат, размер матча, `top-k`, `tie_break_policy`, `code_policy`.
4. Подтверждает старт.
5. Лобби блокируется для входа и выхода, а состав фиксируется из текущих участников лобби.
6. На странице лобби появляется вкладка `Соревнование`.
7. Teacher/admin следит за обновляемой сеткой, матчами, спорными результатами и warnings.
8. При `manual` tie-break принимает решение.
9. При нарушениях применяет moderation action.
10. Когда все игры завершены и победитель выбран, нажимает `Завершить соревнование`.
11. Активная вкладка `Соревнование` пропадает, итоги доступны во вкладке `Архив соревнований`.

Обязательные UX-правила:
- экран соревнования должен явно показывать текущую фазу;
- moderation actions должны быть доступны только teacher/admin;
- warnings антиплагиата не должны автоматически наказывать участника;
- соревнование не должно визуально сводиться к одной таблице, если идет активный event;
- завершение соревнования после определения победителя является явным teacher/admin action, а не только автоматическим terminal state.

#### 3.4.8 Сценарий G: teacher/admin подключает или обновляет игру

Цель пользователя:
- добавить публичный git source;
- запустить sync;
- увидеть validation/build результат;
- понять, активировалась ли новая версия;
- безопасно обработать несовместимость с активными лобби или соревнованиями.

Основной flow:
1. Teacher/admin открывает admin-раздел game sources.
2. Добавляет публичный git URL.
3. Запускает sync вручную.
4. Видит этапы: clone, validation, runtime build, renderer build, policy-check, activation.
5. При успехе видит новую `GameVersion`, commit SHA и image digest.
6. При ошибке видит machine-readable error и build diagnostics.
7. Если есть активное соревнование, sync/activation блокируется.
8. Если есть training lobby, UI показывает результат проверки совместимости slot schema.

Обязательные UX-правила:
- sync должен быть представлен как pipeline, а не как мгновенная кнопка;
- disabled source не может запускать sync и UI обязан это объяснять;
- commit SHA, build id, image digest и error_message должны быть доступны на экране диагностики;
- admin UI может быть более утилитарным, но не должен терять причинно-следственную цепочку.

#### 3.4.9 Сценарий H: admin управляет worker plane

Цель пользователя:
- видеть состояние worker-нод;
- безопасно выводить worker из пула;
- понимать, почему run не стартуют;
- вернуть worker в работу.

Основной flow:
1. Admin открывает worker control.
2. Видит список worker: `online`, `offline`, `draining`, `disabled`.
3. Видит capacity, labels, heartbeat, active leases.
4. Переводит проблемный worker в `draining` или `disabled`.
5. Проверяет очередь scheduler и зависшие run.
6. После стабилизации возвращает worker в `online`.

Обязательные UX-правила:
- `draining` должен объясняться как режим без новых run, но с завершением текущих;
- `disabled` должен выглядеть как явная ручная изоляция;
- worker labels должны быть видны рядом с задачами, которым они нужны;
- UI не должен создавать впечатление, что Redis является source of truth для run lifecycle.

#### 3.4.10 Навигационные следствия для frontend

Новая структура frontend должна группировать страницы вокруг сценариев, а не вокруг таблиц сущностей.

Верхнее меню приложения должно быть коротким и сценарным.

Для ученика видны основные разделы:
1. `Задачи` - каталог задач, single-task run, прогресс, сравнение результатов и документация задач.
2. `Лобби и соревнования` - список открытых лобби, вход в лобби, вкладки `Код` / `Лобби` / `Игра` / `Соревнование` / `Архив соревнований`.
3. `Игры` - список игр с доступом к документации игр.

Для teacher дополнительно внутри этих разделов доступны управляющие действия:
- в `Лобби и соревнования` teacher/admin видит `+` для создания лобби;
- в `Игры` teacher/admin видит `+` для добавления игры и действия изменения настроек игры.

Для admin дополнительно виден раздел:
4. `Статус системы / общие настройки` - состояние сервисов, workers, game sources diagnostics, runtime/system settings и другие административные параметры.

Отдельные разделы `Команды`, `Матчи`, `Повторы` и самостоятельное `Администрирование` не должны быть главными пунктами верхнего меню для ученика. Эти сущности доступны из контекста задач, лобби, игр или системного раздела.

Правила навигации:
- для student-flow multiplayer-игры основным контейнером является страница лобби с вкладками `Код`, `Лобби`, `Игра`;
- для lobby-attached соревнования основным контейнером также остается страница лобби, а не отдельная обязательная страница соревнования;
- пользователь не должен попадать в отдельный сценарий подготовки команды до входа в лобби;
- вкладки должны отражать состояние сценария: незаполненные роли ведут в `Код`, состояние участников/очереди ведет в `Лобби`, найденная или текущая игра ведет в `Игра`, активное соревнование ведет в `Соревнование`;
- раздел `Команды` не должен быть обязательным пользовательским путем для входа в игру, пока действует правило "одна команда пользователя на игру";
- если пользователь приходит из лобби в match viewer, возврат должен вести обратно в это лобби;
- если пользователь приходит из соревнования в match viewer, возврат должен вести обратно в соревнование;
- если пользователь запускает `single_task`, он остается в учебном сценарии, а не проваливается в общий список run;
- run, replay и logs должны быть доступны из контекста исходного сценария;
- страницы низкого уровня (`run`, `match`, `replay`) должны иметь понятный origin context.

#### 3.4.11 Минимальные acceptance criteria для реструктуризации frontend

Frontend-реструктуризация считается успешной, если:
- каждый основной экран можно связать минимум с одним сценарием из этого раздела;
- для `student` доступны понятные пути: решить задачу, выбрать лобби, заполнить код во вкладке `Код`, нажать `Готов` / `Play`, увидеть состояние лобби и очередь, увидеть текущую игру, нажать `Не готов` / `Stop`, вернуться к редактированию кода для следующей игры, участвовать в соревновании, посмотреть replay;
- для `teacher` доступны понятные пути: создать лобби, начать соревнование внутри лобби, провести event, разобрать проблемный run, синхронизировать или настроить игру;
- для `admin` доступны понятные пути: диагностировать game source, worker и execution plane;
- на каждом экране видны: текущий объект, статус объекта, главное действие, причина недоступности действия;
- `viewer` и `editor` являются first-class элементами соответствующих сценариев;
- пользователь не вынужден понимать внутренние сущности `Run`, `Match`, `Replay`, `GameVersion` до момента, когда они нужны для диагностики или разбора результата.

## 4. Ключевые архитектурные решения

### 4.1 Платформа должна поддерживать два режима игр
Поддержка обоих режимов обязательна:
- `single_task`;
- `multiplayer`.

### 4.2 Игра - основная единица интеграции
На первом этапе каждая задача считается отдельной игрой.

В будущем архитектура должна позволять одной игре иметь несколько сценариев или уровней, но в `v1/v2` обязательным минимальным контрактом является подключение новой логики как новой игры.

### 4.3 Команды обязательны
Система должна поддерживать команды.

MVP-модель команды:
- команда создается для конкретной игры;
- у команды есть капитан;
- пока только капитан редактирует коды;
- команда содержит набор кодов по слотам/ролям, определенным игрой;
- команда присоединяется к лобби целиком;
- одна и та же команда может одновременно участвовать в нескольких лобби, если у нее есть доступ;
- в одном конкретном лобби у капитана не может быть одновременно более одной команды.

Для текущего этапа развития продукта действует упрощающее UX- и доменное правило:
- у одного пользователя есть не более одной команды для одной игры;
- команда в этом смысле является не социальной группой, а рабочим набором кодов пользователя для конкретной игры;
- в пользовательском интерфейсе такой объект называется игроком или участником, потому что игрок и команда на текущем этапе фактически одно и то же;
- если игра требует одного агента, команда содержит один слот кода;
- если игра требует несколько агентов/ролей, команда содержит несколько слотов кода;
- при присоединении пользователя к лобби для игры платформа автоматически создает его команду для этой игры, если она еще не существует;
- пользователь участвует в лобби именно этой командой;
- в будущем модель может быть расширена до нескольких команд на пользователя для одной игры, но текущий frontend не должен заставлять пользователя выбирать между командами там, где выбора фактически нет.

### 4.4 Пользовательский код поддерживает два режима API
Платформа должна поддерживать два режима взаимодействия пользовательского кода с игрой:
- `turn_based` - игра вызывает код пошагово;
- `script_based` - пользователь пишет более "сценарный" код через игровой SDK.

Оба режима обязаны работать поверх одного общего игрового рантайма и одного общего процесса оркестрации запусков.

### 4.5 Git-интеграция обязательна
Игры могут храниться:
- в монорепозитории платформы;
- во внешнем публичном git-репозитории, который укажет admin/teacher.

Подключение внешней игры происходит вручную через UI.

### 4.6 Docker-only execution
Весь пользовательский код и код игрового движка выполняются только в Docker.

### 4.7 Масштабирование через удаленные игровые воркеры
Игровые воркеры должны выноситься на отдельные серверы.

### 4.8 Frontend - полноценный SPA
Frontend должен быть переписан как SPA на Vue 3.

### 4.9 Renderer игры загружается гибридно
Платформа предоставляет общую оболочку SPA, а сама игра может поставлять свой renderer.

Для безопасности renderer из внешнего git-репозитория загружается в `iframe`.

## 5. Технологический стек (фиксированный для целевой реализации)

### 5.1 Backend
- Python 3.12+
- FastAPI
- SQLAlchemy 2.x
- Alembic
- Pydantic v2
- Redis
- WebSocket endpoints для realtime

### 5.2 Frontend
- Vue 3
- Vite
- TypeScript
- Pinia
- Vue Router
- Bootstrap 5
- CodeMirror 6 (web code editor with syntax highlight)

### 5.3 Infra
- PostgreSQL 15+
- Redis 7+
- Docker
- docker compose / compose.yml
- Nginx как reverse proxy для production

### 5.4 Почему сохраняется Redis
Redis остается в системе, потому что:
- уже присутствует в текущем проекте;
- подходит для очередей, pub/sub, блокировок, heartbeat и transient-state;
- на текущем этапе его преимуществ достаточно, чтобы не добавлять более тяжелую шину сообщений.

### 5.5 Зафиксированные технологические решения `v2`
Следующие технологические решения считаются согласованными и не должны пересматриваться без отдельной причины:

1. Новый backend строится на `FastAPI`, а не на `Flask`.
2. Realtime transport между backend и SPA строится на обычном `WebSocket`, а не на `Socket.IO`.
3. `Redis` остается coordination layer для очередей, heartbeat, pub/sub и transient-state.
4. Планирование запусков реализуется собственным `scheduler-service`, а не через `Celery`.
5. Сборка игровых пакетов и исполнение игр разделяются на разные сервисы:
   - `builder-service`
   - `game-worker`
6. Внутренняя авторизация веб-приложения после GeekClass login строится на `server-side session` и cookie-based session model.
7. Тестовый стек фиксируется так:
   - backend: `pytest`, `pytest-asyncio`, `httpx`
   - frontend: `vitest`
   - e2e: `playwright`

### 5.6 Что сознательно не используется в `v2`
В `v2` не планируется использовать как базовую часть архитектуры:
- `Socket.IO`
- `Celery`
- `RabbitMQ`
- `Kafka`
- `GraphQL`
- `Kubernetes`

Если эти технологии когда-либо появятся позже, это должно быть оформлено как отдельное архитектурное решение, а не как "тихая" подмена стека.

## 6. Качество реализации и обязательные инженерные требования

### 6.1 DDD обязателен
Код должен строиться по DDD-подходу.

Требования:
- разделение на bounded contexts;
- доменная логика находится в `domain`, а не в роутерах и не в ORM-сущностях контроллерного слоя;
- application layer оркестрирует use-case'ы;
- infrastructure layer содержит БД, git, Docker, Redis, websocket transport, внешние интеграции;
- явные domain entities, value objects, repositories, services;
- инварианты агрегатов реализуются в доменной модели, а не размываются по коду.

### 6.2 Слои backend
Для каждого bounded context использовать структуру вида:

```text
backend/src/<context_name>/
  domain/
  application/
  infrastructure/
  api/
```

Общий слой:

```text
backend/src/shared/
  kernel/
  db/
  events/
  security/
  utils/
```

### 6.3 Правила чистоты кода
- строгая типизация в Python;
- без "god services" и без гигантских файлов;
- без бизнес-логики в HTTP endpoints;
- без прямого смешивания ORM-моделей и DTO фронтенда;
- без молчаливых `except:` в бизнес-коде;
- без скрытых side effect в моделях;
- все важные переходы состояния должны быть явно отражены в коде и тестах.

### 6.4 Обязательные тесты
Тесты обязательны.

Минимальный набор:
- unit tests для доменной логики;
- integration tests для БД, репозиториев, worker orchestration, git sync, Docker runner;
- API tests;
- contract tests для игровых пакетов;
- e2e smoke tests для ключевых UI-сценариев.

### 6.5 Definition of Done для любой новой фичи
Фича считается выполненной только если:
- есть код;
- есть тесты;
- соблюдены инварианты домена;
- обновлена OpenAPI-схема, если менялся API;
- обновлены пользовательские ошибки и статусы;
- есть логирование для критичных путей;
- нет регрессии базовых сценариев.

## 8. Роли и права

### 8.1 `student`
Может:
- входить в систему;
- создавать команды для доступных игр;
- редактировать код своей команды, если он капитан;
- входить в открытые лобби;
- входить в лобби по коду;
- становиться `ready`;
- запускать одиночные задачи;
- смотреть свои результаты;
- смотреть публичные live/replay, если это разрешено игрой и лобби.

### 8.2 `teacher`
Может все, что `student`, и дополнительно:
- создавать и закрывать лобби;
- создавать соревнования;
- синхронизировать игры из git;
- включать и выключать публичный spectator mode;
- смотреть warnings по схожести решений;
- переводить участника/команду в `not_ready`;
- временно банить участника/команду;
- управлять доступом по коду;
- просматривать технические логи матчей;
- останавливать активные тренировочные матчи и лобби.

### 8.3 `admin`
Может все, что `teacher`, и дополнительно:
- управлять системными настройками;
- управлять runtime profiles;
- управлять глобальными параметрами безопасности;
- удалять игры и git-источники;
- просматривать и настраивать воркеры.

### 8.4 Источник прав
Роль пользователя берется из GeekClass JWT как есть.

Дополнительного локального маппинга ролей в `v2` не требуется.

## 11. Инварианты домена

### 11.1 Игра
- у игры всегда есть активная версия;
- игра не может быть доступна пользователям без валидной версии;
- игра не может быть синхронизирована во время активного соревнования по этой игре;
- если у игры есть активные тренировочные лобби, синхронизация с несовместимым изменением схемы обязательных слотов запрещена.

### 11.2 Команда
- команда принадлежит ровно одной игре;
- слот команды должен соответствовать слотам, объявленным активной версией игры;
- команда не может перейти в `ready`, если не заполнены все обязательные слоты.

### 11.3 Тренировочное лобби
- teacher/admin - единственные, кто может создавать лобби;
- команда может войти в лобби даже без полного набора кодов;
- команда не может стать `ready`, пока обязательные слоты пусты;
- snapshot кода фиксируется при создании/постановке run в очередь;
- если код изменился после фиксации snapshot текущего батча, изменения применяются только в следующем батче;
- лобби живет, пока teacher/admin не остановит, не закроет или не удалит его;
- teacher/admin может настроить срок автоудаления старых terminal training-match run;
- при обновлении игры все команды в `ready` обязаны пройти повторную валидацию относительно новой `GameVersion`;
- `ready` может быть сохранен только если новая версия совместима по схеме обязательных слотов и повторная валидация успешно пройдена.

### 11.4 Соревнование
- во время активного соревнования синхронизация игры запрещена;
- состав участников фиксируется по правилам соревнования;
- бан участника или команды во время соревнования должен прервать текущие матчи этой сущности;
- порядок трактовки прерванных матчей задается `ban_resolution_policy`.

### 11.5 Одиночная задача
- пользователь не может иметь более одного активного запуска по всей платформе;
- ограничение активного запуска проверяется по `user_id`, даже если попытка исполняется от имени команды или однослотовой команды;
- запуск новой попытки невозможен, пока предыдущая не завершена или не остановлена;
- история попыток сохраняется;
- лидерборд показывает лучший результат, но хранится вся история.

### 11.6 Worker plane
- `Run` может быть назначен только на `WorkerNode` со статусом `online`;
- worker в состоянии `draining` не получает новые `Run`, но может завершать уже запущенные;
- worker в состоянии `offline` или `disabled` не получает новые `Run`;
- source of truth по статусам запусков остается у control plane, а не у worker.

## 12. Игровой пакет и структура репозитория игры

Каждая игра обязана удовлетворять стандартному layout:

```text
game-repo/
  manifest.yaml
  README.md
  backend/
    Dockerfile
    src/
  renderer/
    package.json
    src/
  templates/
    python/
  docs/
    player_guide_ru.md
    sdk_guide_ru.md
```

### 12.1 Обязательные части
- `manifest.yaml`
- backend runner
- frontend renderer
- шаблоны кода
- README
- документация для игрока/задачи;
- инструкция по SDK, если игра `script_based`

### 12.2 Минимальный `manifest.yaml`
```yaml
id: tanks_ctf_v1
title_ru: Танки: захват флага
version: 1.0.0
protocol_version: 1
game_mode: multiplayer
min_players_per_match: 2
max_players_per_match: 16
code_api_mode: turn_based
spectator_mode_supported: true
leaderboard_enabled: false
scoreboard_mode: score
team_slots:
  - slot_key: runner
    title_ru: Захватчик
    required: true
    template_ref: templates/python/runner.py
  - slot_key: defender
    title_ru: Защитник
    required: true
    template_ref: templates/python/defender.py
runtime:
  language: python
  image: local-build
renderer:
  entry: renderer/dist/index.html
training_defaults:
  max_steps: 1000
  turn_timeout_ms: 100
competition_defaults:
  max_steps: 1500
  turn_timeout_ms: 80
```

Правила `game_mode`:
- новые игровые пакеты используют только `single_task` или `multiplayer`;
- `multiplayer` обязан явно задавать `min_players_per_match` и `max_players_per_match`;
- `single_task` не должен задавать `min_players_per_match`/`max_players_per_match`;
- `2 <= min_players_per_match <= max_players_per_match <= 64`;
- legacy manifests со значениями `small_match` и `massive_lobby` допустимы только для обратной совместимости и нормализуются в `multiplayer`;
- для legacy `small_match` дефолтный диапазон равен `2..2`, для legacy `massive_lobby` - `2..64`, если явные значения не указаны.

### 12.3 Игровой пакет обязан описывать
- режим игры;
- режим пользовательского API;
- список обязательных и необязательных слотов команды;
- `required_worker_labels` (опционально), если игра требует запуск в определенном пуле worker-нод;
- шаблоны кода по слотам;
- базовые параметры тренировки;
- базовые параметры соревнования;
- поддерживает ли игра spectator mode;
- есть ли лидерборд;
- имеет ли игра score.

### 12.4 Источник метаданных из игрового пакета
Игровой пакет обязан поставлять:
- технические метаданные;
- README;
- инструкции по SDK;
- документацию игрока/задачи, которая является частью содержимого игры;
- шаблоны кода.

Для `single_task` и других режимов используется двухслойная модель source of truth:
- технический контракт игры (manifest, slot schema, runtime profile, renderer contract, protocol flags) всегда приходит из игрового пакета;
- каталожные поля для витрины обучения (человекочитаемое описание, сложность, темы) хранятся в UI/БД платформы.

Если игра из репозитория поставляет human-readable описание, оно может использоваться только как seed для формы в админке.
Окончательная precedence policy для каталожных полей зафиксирована в разделе `20.1.2`.

Документация игры/задачи не считается внешней справкой. Для пользователя она является частью задачи: объясняет правила мира, доступные действия, роли, SDK и критерии результата. UI обязан открывать ее из контекста задачи, лобби или вкладки `Код`, сохраняя связь с текущей игрой.

## 13. Режимы пользовательского API

### 13.1 `turn_based`
Игровой движок вызывает код участника пошагово.

Поддерживаемые варианты entrypoint:
- модульная функция `make_move(observation)`;
- класс `Bot` с методом `make_move`.

Разрешено:
- хранить состояние между ходами в памяти процесса;
- использовать глобальные переменные или состояние экземпляра класса.

Рекомендуемый шаблон:
```python
class Bot:
    def __init__(self):
        self.memory = {}

    def on_match_start(self, context):
        self.memory["map_size"] = context["map_size"]

    def make_move(self, observation):
        return {"action": "wait"}
```

### 13.2 `script_based`
Для задач в стиле CodeCombat участник пишет "сценарный" код через SDK:
- без `input()`;
- с доступом к объекту SDK;
- с возможностью писать `while` и последовательность действий.

Пример:
```python
from sdk import hero

while hero.can_continue():
    if hero.see_coin():
        hero.move_to_coin()
    else:
        hero.turn_right()
```

### 13.3 Управляемое исполнение `script_based`
Платформа обязана исполнять `script_based` код управляемо:
- не давать пользовательскому коду бесконтрольно захватывать процесс навсегда;
- прерывать исполнение после каждого игрового действия SDK;
- продолжать выполнение только под контролем движка;
- вести историю шагов и логов.

Внутренняя реализация может использовать generator/coroutine/cooperative runtime, скрытый от пользователя.

### 13.3.1 Нормативная модель исполнения `script_based`
Для целей реализации и кодогенерации `script_based` трактуется так:
- пользователь пишет обычный Python-код, который выглядит как непрерывный сценарий;
- источником взаимодействия с миром являются только объекты SDK, предоставленные игрой;
- каждый mutating SDK-вызов или явный `sdk.wait()` считается одним игровым действием;
- после каждого игрового действия выполнение пользовательского сценария обязано быть приостановлено;
- при следующем тике движок обязан продолжить выполнение с точной следующей точки пользовательской программы;
- локальные переменные, глобальные переменные и состояние объектов пользователя сохраняются между такими возобновлениями до завершения запуска.

### 13.3.2 Что обязана гарантировать реализация
Реализация `script_based` runtime обязана гарантировать:
- детерминированную точку возобновления после каждого игрового действия;
- ограничение времени вычислений между двумя соседними игровыми действиями;
- невозможность бесконечного CPU-loop без SDK-вызова;
- корректное завершение сценария, если пользовательский код дошел до конца;
- перевод запуска в `timeout` или `failed`, если между двумя игровыми действиями превышен лимит времени или лимит инструкций.

### 13.3.3 Поддерживаемые конструкции Python в `script_based`
Гарантированно поддерживаются:
- переменные;
- условия;
- циклы;
- функции;
- простые классы;
- списки, словари, множества, кортежи;
- арифметика, строки, базовые модули из allowlist runtime profile;
- обработка исключений в пределах одного пользовательского сценария.

### 13.3.4 Что не гарантируется и должно считаться запрещенным
В `script_based` не должны поддерживаться как часть публичного контракта:
- `exec` и `eval`;
- `async`/`await`;
- потоки и процессы;
- monkey patching SDK-объектов;
- интроспекция стека исполнения и code objects;
- динамический импорт модулей, не разрешенных runtime profile;
- любые механики, позволяющие обойти cooperative runtime модели.

### 13.3.5 Формальный lifecycle одного тика `script_based`
Нормативный runtime lifecycle для одного тика:
1. движок возобновляет пользовательский сценарий из последней детерминированной точки;
2. сценарий выполняется до первого терминального события:
   - mutating SDK-вызов;
   - `sdk.wait()`;
   - завершение пользовательской программы;
   - необработанное исключение;
   - превышение лимита инструкций/времени между SDK-действиями;
3. движок фиксирует результат этого тика (action intent / completion / error / timeout);
4. движок приостанавливает сценарий и возвращает управление orchestration-слою.

Обязательное следствие:
- двумя соседними тиками считается участок исполнения между двумя соседними точками cooperative-yield;
- этот участок всегда ограничивается `inter_action_cpu_budget` и `inter_action_wall_time_budget`.

### 13.4 `input()` и `print()`
- `input()` запрещен;
- платформа валидирует пользовательский код при сохранении слота и отклоняет `input()` с `422 invariant_violation`, не дожидаясь запуска;
- `print()` разрешен только как отладочный вывод;
- отладочный вывод виден только автору попытки и teacher/admin;
- отладочные логи не видны зрителям.

### 13.5 Стандартная библиотека Python
Разрешается почти вся стандартная библиотека Python, кроме опасных модулей и API, нарушающих модель песочницы.

Обязательная denylist для `v2`:
- `subprocess`
- `socket`
- `asyncio.subprocess`
- `multiprocessing`
- `ctypes`
- прямые process-control API

Финальный denylist можно расширять настройками runtime profile.

## 15. Git-синхронизация игр

### 15.1 Добавление игры из git
Teacher/admin добавляет игру через UI, указывая:
- URL публичного git-репозитория.

### 15.2 Синхронизация
Синхронизация запускается только вручную.

Алгоритм:
1. скачать или обновить репозиторий;
2. взять `HEAD` default branch;
3. считать `manifest.yaml`;
4. провалидировать структуру пакета;
5. собрать runtime image в изолированном Docker build;
6. собрать renderer bundle;
7. создать новую `GameVersion`;
8. сделать новую версию активной.

### 15.2.1 Безопасность pipeline синхронизации
Синхронизация и сборка игры из git обязаны выполняться в отдельном изолированном builder environment.

Обязательные требования:
- builder не работает в процессе `api-service`;
- builder не имеет доступа к секретам платформы, production env, session storage и базе данных платформы;
- build context ограничен только checkout-копией конкретного репозитория игры;
- в build pipeline нельзя пробрасывать хостовые директории с кодом платформы;
- кэш сборки, если используется, не должен содержать секреты и должен быть отделен от runtime среды платформы;
- сетевой доступ builder ограничивается только необходимыми внешними адресами для clone и разрешенных package registries;
- builder исполняется в отдельном rootless-runtime (или эквивалентной безпривилегированной песочнице), а не в доверенном privileged-контуре платформы;
- build-этап должен стартовать с политикой deny-by-default по сети; доступ в сеть открывается только на allowlist endpoints (git host и разрешенные registries);
- артефакты build (runtime image digest, renderer artifact hash) фиксируются как immutable и проходят policy-check до активации версии;
- если validation, build или security checks не пройдены, текущая активная версия игры не меняется;
- активация новой `GameVersion` разрешена только после полного прохождения validation и build этапов.

### 15.3 Правила обновления активной игры
Если идет активное соревнование по игре:
- синхронизация запрещена.

Если по игре активны тренировочные лобби:
- перед переключением версии выполняется проверка совместимости slot schema;
- совместимым считается обновление, которое не удаляет существующие обязательные слоты, не переименовывает их и не повышает необязательный слот до обязательного для уже существующих лобби;
- добавление новых необязательных слотов допускается;
- если схема слотов несовместима, sync завершается ошибкой `GAME_SYNC_INCOMPATIBLE_SLOT_SCHEMA`, а активная версия не меняется;
- в `v2` режим принудительного переключения на несовместимую slot schema не поддерживается;
- все текущие матчи принудительно останавливаются;
- их статус становится `canceled`, а `error_message` получает reason-code `canceled_by_game_update`;
- результаты не попадают в лидерборд;
- лобби перезапускаются на новой версии игры;
- команды остаются в лобби;
- каждая команда проходит повторную валидацию против новой версии;
- статус `ready` сохраняется только у команд, прошедших повторную валидацию.

### 15.4 Версионирование
Каждая синхронизация создает новую неизменяемую версию игры.

Даже если игра "сразу переключается", commit SHA должен храниться и быть видимым в UI и аудите.

## 16. Команды и коды команды

### 16.1 Команда
Команда создается для конкретной игры.

У команды:
- один капитан;
- набор слотов;
- коды по слотам;
- история снимков кода.

### 16.2 Слоты команды
Слоты и их обязательность задаются игрой.

Примеры:
- `runner`
- `defender`
- `carrier`
- `scout`

### 16.3 Шаблоны кода
Игра сама поставляет шаблоны кода по слотам.

### 16.4 Снимок кода для матча
Перед стартом матча создается неизменяемый snapshot:
- по одному snapshot на слот;
- snapshot привязывается к матчу или попытке;
- дальнейшие изменения кода не влияют на уже стартовавший запуск.

### 16.4.1 Момент фиксации snapshot
Для целей реализации момент фиксации snapshot определяется строго:
- для `Match` это переход `created -> queued`;
- для `SingleTaskAttempt` это переход `created -> queued`.

В рамках одной транзакции система обязана:
- создать недостающие `CodeSnapshot`;
- записать идентификаторы snapshot в `Run.input_payload_json` или эквивалентную неизменяемую структуру входа запуска;
- перевести матч или попытку в `queued`.

После завершения этой транзакции никакие последующие изменения командного кода не могут влиять на конкретный `Match` или `SingleTaskAttempt`.

### 16.5 Изменение кода в очереди
До момента перехода `created -> queued` матч или попытка могут быть пересобраны на основе более нового кода команды.

После перехода в `queued` используется уже зафиксированный snapshot, даже если worker еще не начал фактическое исполнение.

Изменение кода команды само по себе не должно автоматически снимать команду с `ready`.

## 17. Тренировочные лобби

### 17.1 Создание лобби
Лобби всегда создает только `teacher` или `admin`.

При создании и последующем редактировании лобби teacher/admin может задавать:
- название;
- игру и активную версию игры на момент создания;
- режим доступа (`public|code`) и код входа;
- лимит участников;
- `auto_delete_training_runs_days` - срок автоматического удаления старых завершенных тренировочных матчей этого лобби.

### 17.2 Типы доступа
Поддерживаются:
- открытое лобби;
- лобби по коду.

### 17.3 Жизненный цикл
Статусы:
- `draft`
- `open`
- `running`
- `paused`
- `stopped`
- `updating`
- `closed`

Смысл статусов:
- `open` - лобби принимает готовых игроков и может запускать новые батчи матчей;
- `running` - в лобби есть активный батч в фазе симуляции/replay/result;
- `paused` - teacher/admin временно приостановил запуск новых матчей без очистки лобби;
- `stopped` - teacher/admin остановил лобби: активные training run отменены, ready-state очищен, новые матчи не стартуют до возврата в `open`;
- `updating` - идет переключение версии игры и повторная валидация команд;
- `closed` - terminal-состояние старого сценария закрытия; новое полное удаление выполняется отдельной командой удаления.

### 17.4 Ready-state
Вход в лобби не означает готовность.

Команда может быть:
- `not_ready`
- `ready`
- `blocked`

### 17.5 Ограничение готовности
Команда не может стать `ready`, если не заполнены все обязательные слоты.

### 17.6 Пул матчей
Внутри одного лобби может параллельно идти несколько матчей.

Матчи запускаются батчами:
- backend выбирает всех готовых свободных участников;
- делит их на максимально возможное число матчей с размером в диапазоне `min_players_per_match..max_players_per_match`;
- если готовых участников недостаточно для еще одного матча, они остаются в очереди;
- все матчи одного батча создаются почти одновременно и визуально группируются в UI по `batch_id`/времени запуска;
- следующий батч не запускается, пока предыдущий батч не завершил фазы `simulation`, `replay` и `result`.

Минимальная скорость replay-периода в лобби: 0,5 секунды на кадр. Если в батче несколько матчей разной длины, длительность replay-периода считается по самому длинному replay.

Фазы цикла лобби:
- `waiting_players` - недостаточно готовых игроков;
- `waiting_viewer` - игроки есть, но нет готового online-viewer для просмотра матча;
- `simulation` - run выполняются на worker plane;
- `replay` - участники смотрят синхронизированный replay;
- `result` - короткое отображение победителя перед следующим циклом.

`LobbyLiveView` должен отдавать UI текущую фазу, человеко-читаемый label/message, `replay_started_at`, `replay_until`, `result_until`, `cycle_frame_ms`, текущие и архивные группы матчей.

### 17.7 Команда в нескольких лобби
Одна команда может участвовать в нескольких лобби одновременно, если у нее есть доступ.

### 17.8 Выход из тренировочного лобби
Если матч еще не идет, команда просто покидает лобби.

Если команда нажимает `Stop` во время активного тренировочного `multiplayer`-матча:
- ее текущий запуск отменяется;
- команда переходит в статус подготовки и не попадает в следующий матч, пока снова не нажмет `Play`.

### 17.9 Управление и удаление лобби
Teacher/admin может:
- перевести лобби в `open`;
- поставить лобби на `paused`;
- остановить лобби (`stopped`);
- изменить настройки лобби;
- удалить лобби полностью.

Удаление лобби:
- требует teacher/admin;
- запрещено, пока к лобби прикреплено активное или ожидающее ручного завершения соревнование;
- удаляет само лобби и связанные training-match run/replay;
- не удаляет глобальную команду пользователя для игры, потому что команда является canonical workspace игрока для этой игры.

Автоудаление старых тренировочных матчей:
- настраивается через `auto_delete_training_runs_days`;
- применяется только к terminal training-match run (`finished|failed|timeout|canceled`);
- не должно удалять run текущего батча, replay которого еще может показываться пользователям;
- при удалении run должны удаляться связанные replay-данные.

## 18. Соревнования

### 18.1 Соревнование - отдельная сущность
Соревнование не является просто тренировочным лобби.

У соревнования есть:
- собственный lifecycle;
- собственный набор entrants;
- собственные настройки;
- собственная логика продвижения.

При этом для текущего UX соревнование может быть прикреплено к конкретному тренировочному лобби.

В таком режиме:
- teacher/admin запускает соревнование из страницы лобби;
- состав соревнования фиксируется из текущих участников лобби;
- лобби блокируется для входа новых участников;
- текущие участники не могут выйти из лобби до завершения соревнования;
- на странице лобби появляется плашка `Идет соревнование`;
- на странице лобби появляется вкладка `Соревнование` с realtime-сеткой или подходящим турнирным представлением;
- текущая игра пользователя по-прежнему отображается во вкладке `Игра`;
- после завершения всех игр и выбора победителя teacher/admin должен нажать `Завершить соревнование`;
- после этого активная вкладка `Соревнование` скрывается;
- если в лобби было хотя бы одно соревнование, появляется вкладка `Архив соревнований` с итогами прошлых соревнований.

Доменная сущность `Competition` остается отдельной, но пользовательский вход в нее для lobby-attached режима идет через `TrainingLobby`.

Матчи соревнования запускаются этапами:
- все матчи текущего раунда/этапа, у которых известны участники, создаются параллельно;
- следующий этап не формируется до завершения всех матчей предыдущего этапа и применения правил продвижения;
- для каждого этапа UI показывает текущие матчи, replay/result состояние, победителей и статистику участников;
- вкладка `Игра` показывает текущий матч пользователя, а вкладка `Соревнование` показывает весь этап.

### 18.2 Форматы
В текущей версии пользовательский и API-сценарий соревнований поддерживает только `single_elimination`.

`round_robin` и `swiss` остаются будущими форматами. Они не должны быть доступны в v2 UI и не должны приниматься backend-контрактом до появления полноценной модели туров, standings и правил подсчета очков.

Для `v2` обязательным является:
- визуализация elimination-bracket;
- rounds/matches view для случаев, где классическая бинарная сетка плохо читается.

### 18.3 Матчи с `N` участниками
Соревнование обязано поддерживать матчи с числом команд больше 2.

### 18.4 Правило продвижения
Правило продвижения настраивается.

Примеры:
- дальше проходит только победитель;
- дальше проходят `top-2`;
- дальше проходят `top-k`.

### 18.4.1 Политика разрешения ничьих
У соревнования обязана быть явно задана `tie_break_policy`.

Поддерживаемые варианты:
- `manual` - соревнование приостанавливается для teacher/admin decision;
- `shared_advancement` - все участники, разделившие граничное место, проходят дальше;
- `tiebreak_match` - будущая политика, недоступна в v2 UI/API до появления автоматических дополнительных матчей;
- `game_defined` - будущая политика, недоступна в v2 UI/API до появления явного game-defined контракта для неоднозначных границ.

### 18.5 Политика обновления кода в соревновании
Настройка соревнования определяет, когда можно менять код:
- `locked_on_registration`
- `locked_on_start`
- `allowed_between_matches`

### 18.6 Бан во время соревнования
Teacher/admin может:
- перевести entrant в `not_ready`;
- забанить entrant.

Если entrant забанен во время активного соревнования:
- все его текущие матчи прерываются;
- трактовка результатов определяется `ban_resolution_policy`.

### 18.7 Проверка схожести решений
Проверка схожести работает только в рамках соревнования.

Она запускается автоматически при каждом новом используемом snapshot в соревновании.

Автоматического наказания нет.

Teacher/admin получает warning.

### 18.8 Завершение соревнования
Когда все игры соревнования завершены и победитель определен, соревнование переходит в статус `completed`.

Статус `completed` означает, что сетка и результаты еще показываются в активной вкладке соревнования, лобби остается заблокированным для входа и выхода, а учитель/admin должен явно нажать `Завершить соревнование`.

Только после этого соревнование получает статус `finished`, вкладка активного соревнования исчезает, а результат становится доступен в архиве соревнований лобби.

## 19. Матчи и попытки

### 19.1 Общий lifecycle матча
Статусы:
- `created`
- `queued`
- `running`
- `finished`
- `failed`
- `canceled`
- `aborted`

### 19.2 Lifecycle одиночной попытки
Статусы:
- `created`
- `queued`
- `running`
- `finished`
- `stopped`
- `failed`
- `timeout`
- `canceled`

### 19.2.1 Причины отмены run (`error_message`)
Для статуса `canceled` API возвращает машинно-читаемый `error_message` с кодом причины.

В `v2` фиксируется следующий словарь кодов:
- `manual_cancel` - ручная остановка через `POST /runs/{runId}/cancel`;
- `manual_stop_single_task` - ручная остановка single-task попытки через `POST /single-tasks/{gameId}/stop`;
- `manual_moderation_ban` - отмена из-за модерационного бана участника в активном соревновании;
- `canceled_by_game_update` - отмена из-за обновления версии игры в training-лобби.
- `lobby_stopped` - teacher/admin остановил training-лобби;
- `lobby_closed` - training-лобби переведено в terminal closed-state;
- `lobby_deleted` - training-лобби удалено вместе с тренировочными матчами.

Правила:
- backend может добавлять дополнительные коды без breaking change;
- неизвестные коды должны отображаться в UI как "служебная причина" с показом raw-значения;
- `error_message` не используется как человеко-читаемый текст ошибки для локализации, а только как стабильный reason-code.

### 19.3 Результат игры
Игра возвращает result payload.

Минимальный контракт:
```json
{
  "status": "finished",
  "winner_ids": ["team-1"],
  "scores": {
    "team-1": 15,
    "team-2": 8
  },
  "placements": {
    "team-1": 1,
    "team-2": 2
  },
  "team_stats": {},
  "player_stats": {},
  "replay_ref": "replay-uuid",
  "metrics": {
    "duration_ms": 25000,
    "ticks": 420
  }
}
```

Правила обязательности полей:
- для `single_task` обязательны `status`, `metrics`, `replay_ref` при наличии реплея;
- для тренировочных multiplayer-матчей обязательны `status` и хотя бы один из блоков `scores` или `placements`;
- для соревновательных матчей обязательны `status`, `placements` и `metrics`;
- если игра возвращает только `scores` для соревновательного матча и не возвращает `placements`, такой результат считается невалидным и матч не может быть использован для турнирного продвижения.

### 19.3.1 Политика равенства результатов
Если у нескольких участников одинаковый `score`, игра обязана:
- либо явно вернуть одинаковые `placements` и тем самым зафиксировать ничью;
- либо вернуть уже разрешенные детерминированные `placements`.

Платформа не должна самостоятельно выводить турнирные места из `scores`, если `placements` отсутствуют.

### 19.4 Что является источником истины для результата
Источником истины является payload, возвращенный игрой.

Платформа не должна вычислять победителя за игру, если игра сама уже вернула placements и scores.

Для соревнований источником истины для продвижения являются именно `placements` с учетом `tie_break_policy`.

## 20. Single-player задачи

### 20.1 Метаданные задачи
Для `single_task` обязательны:
- описание;
- документация задачи/игры;
- сложность;
- темы.

Для `v2` источником истины для этих полей является UI/БД платформы, а не manifest репозитория.

### 20.1.1 Onboarding single-task игры после sync
Если новая `single_task` игра синхронизирована из репозитория:
- платформа создает техническую запись игры и версии;
- README, `player_guide_ru.md`/`docs/player_guide_ru.md` и `sdk_guide_ru.md`/`docs/sdk_guide_ru.md`, если существуют, используются как исходный материал для первичного заполнения формы и документации задачи;
- игра получает `catalog_metadata_status = draft`, пока teacher/admin не заполнит обязательные поля каталога в UI;
- до перевода `catalog_metadata_status` в `ready` такая игра не должна отображаться в публичном каталоге задач.

Текущая реализация MVP:
- `catalog_metadata_status` возвращается в `GameResponse`;
- для `single_task` регистрация без `description/difficulty/topics` создает игру в `draft`;
- endpoint `GET /catalog/single-tasks` возвращает только `single_task` игры со статусом `ready`.

### 20.1.2 Precedence policy для каталожных полей
При конфликте между репозиторием и UI/БД:
- описание задачи, сложность и темы берутся из UI/БД;
- документация задачи, README, SDK guide, шаблоны и технические параметры берутся из игрового пакета, если teacher/admin не отредактировал опубликованную документацию задачи в UI;
- UI/БД может override только каталожные поля, а не технический контракт игры.

### 20.2 Темы
Одна задача может относиться к нескольким темам.

Примеры:
- графы;
- BFS/DFS;
- жадные алгоритмы;
- динамическое программирование;
- симуляция.

Каталог задач обязан поддерживать группировку по темам.

Если задача относится к нескольким темам, реализация должна явно выбрать и задокументировать один из вариантов:
- показывать задачу в основной теме и отображать остальные темы как tags;
- показывать задачу в нескольких тематических группах с дедупликацией в summary/progress;
- показывать матричную группировку, где тема является главным разделом, а сложность - подгруппой.

### 20.3 Сложность
Сложность хранится как минимум в одном из значений:
- `easy`
- `medium`
- `hard`

Каталог задач обязан поддерживать группировку по сложности.

Минимально допустимые варианты:
- секции `easy`, `medium`, `hard` внутри выбранной темы;
- группировка по теме с подгруппами сложности;
- группировка по сложности с вложенными темами.

Плоский список карточек допустим только как результат активного поиска или фильтра, но не как основной вид каталога задач.

### 20.4 Лидерборд
Если у задачи включен leaderboard:
- хранится история всех попыток;
- в таблицу попадает лучший результат пользователя.

Если у задачи нет score-модели:
- можно показывать список решивших пользователей.

### 20.5 Глобальная метрика решенных задач
Платформа должна уметь показывать, кто решил больше задач.

## 21. Spectator mode и реплеи

### 21.1 Два режима просмотра
Поддерживаются:
- live-view;
- replay после завершения.

### 21.2 Публичность
В текущем MVP просмотр run/replay требует сессию. Доступ разрешается участнику матча, участнику соответствующего лобби/соревнования, teacher или admin.

Публичный spectator link является отдельным будущим режимом и должен включаться только явной настройкой лобби/соревнования.

### 21.3 Отображение на уровне лобби
Страница лобби или соревнования показывает список матчей.

Зритель выбирает конкретный матч и открывает его.

### 21.4 Безопасность renderer
Renderer из стороннего репозитория:
- работает в `iframe`;
- не получает auth tokens пользователя;
- не обращается к приватным API напрямую.

Оболочка SPA:
- получает данные сама;
- передает renderer'у только разрешенные события и состояния через `postMessage`.

### 21.5 Что передает shell renderer'у
- начальную конфигурацию игры;
- поток кадров;
- события матча;
- итоговый result payload;
- данные реплея.

### 21.6 Реплеи
Реплей должен хранить:
- кадры;
- события;
- финальный summary;
- версию игры;
- commit SHA игры;
- конфигурацию запуска.

Эти данные могут храниться либо прямо в записи `Replay`, либо быть детерминированно восстанавливаемыми через связку `Replay -> Match -> GameVersion`, но API реплея обязан отдавать их как часть единого ответа без дополнительных догадок клиента.

## 22. Авторизация GeekClass

Для `v2` авторизация переносится 1-в-1 по аналогии с `GeekCodeBattle`.

### 22.1 Login flow
1. backend делает redirect на `{GEEKCLASS_HOST}/insider/jwt?redirect_url={callback_url}`;
2. GeekClass возвращает JWT в query-параметре `token`;
3. backend декодирует JWT по `JWT_SECRET`, `HS256`;
4. backend проверяет `iat`, TTL токена - 60 секунд;
5. backend делает upsert локального пользователя;
6. backend сохраняет вход в server-side session.

### 22.1.1 Session model
Внутренний auth-state платформы после успешного входа обязан храниться как server-side session:
- браузер получает session cookie;
- cookie должна быть `HttpOnly`;
- cookie должна иметь безопасные флаги для production (`Secure`, подходящий `SameSite`);
- SPA не хранит собственный long-lived access token для основной работы приложения;
- backend проверяет сессию на каждом защищенном запросе API и WebSocket handshake.

### 22.2 JWT payload
Ожидаемый формат:
```json
{
  "id": 123,
  "role": "student",
  "name": "Иван Иванов",
  "iat": 1713272000
}
```

### 22.3 Dev login
Для локальной разработки обязателен `dev-login`.

Dev login должен позволять:
- войти под любой ролью;
- задать произвольный nickname;
- задать внешний id.

## 23. Песочница и исполнение

### 23.1 Обязательные ограничения Docker
- без `--privileged`;
- без доступа к Docker socket;
- без доступа к хостовой БД;
- изолированная файловая система;
- временная рабочая директория;
- автоматическое удаление контейнера после завершения;
- лимиты CPU, RAM, PIDs;
- лимит времени;
- сеть настраивается на уровне игры.

### 23.2 Коды участников и игровые движки
Исполняются изолированно:
- код участника - в изолированном runtime;
- игровой движок - в отдельном Docker-контейнере игры;
- платформа не исполняет game code в своем собственном процессе.

### 23.3 Worker model
Компоненты:
- `api-service`
- `scheduler-service`
- `builder-service`
- `game-worker`
- `redis`
- `postgres`

Архитектурно система делится на:
- `control plane`: `api-service`, `scheduler-service`, `postgres`, `redis`
- `build plane`: `builder-service`
- `execution plane`: множество `game-worker`, которые могут располагаться как локально, так и на удаленных серверах.

### 23.4 Планировщик
Планировщик:
- выбирает свободный worker;
- ставит job в очередь;
- отслеживает heartbeat;
- обновляет статус запуска;
- обрабатывает таймауты.

Дополнительно планировщик обязан:
- быть единственным сервисом, который принимает решение о назначении `Run` на конкретный worker;
- использовать как источник истины `PostgreSQL` для жизненного цикла `Run`;
- использовать `Redis` как coordination layer, а не как долговременный source of truth;
- уметь работать с worker, находящимися на удаленных хостах.

### 23.5 Heartbeat
Каждый worker периодически сообщает:
- свою доступность;
- число свободных слотов;
- активные запуски;
- текущую нагрузку.

Минимальный heartbeat payload должен включать:
- `worker_id`
- `hostname`
- `available_slots`
- `running_run_ids`
- `supported_runtime_profiles`
- `last_seen_at`
- агрегированную информацию по CPU/RAM

### 23.6 `builder-service`
`builder-service` отвечает только за build/sync pipeline игровых пакетов:
- clone/update git-репозитория игры;
- validation структуры пакета;
- build runtime image;
- build renderer artifact;
- публикацию или подготовку immutable артефактов для исполнения.

`builder-service` не должен:
- исполнять пользовательские матчи;
- участвовать в realtime матчей;
- совмещаться с `api-service` в одном процессе.

### 23.7 `game-worker`
`game-worker` отвечает только за исполнение уже подготовленных запусков.

`game-worker` обязан:
- получать задачу исполнения после назначения от `scheduler-service`;
- запускать только готовые runtime image по immutable image reference / digest;
- не выполнять build из git-репозитория во время матчевого исполнения;
- регулярно отправлять heartbeat;
- публиковать lifecycle events и игровые события обратно в control plane;
- корректно завершать и очищать контейнеры после завершения `Run`.

### 23.8 Remote workers
`game-worker` может размещаться:
- на том же хосте, что и control plane;
- на отдельном сервере в той же сети;
- на нескольких удаленных серверах для горизонтального масштабирования.

С точки зрения платформы новый сервер масштабирования - это просто еще один зарегистрированный `game-worker`.

### 23.8.1 Worker labels и операционный профиль
Для удаленных worker-нод используется конфигурируемый набор меток (`worker labels`), например:
- `region`;
- `pool`;
- `hardware_class`.

`worker labels`:
- регистрируются вместе с worker в control plane;
- используются как metadata для observability и операционного управления;
- участвуют в scheduler-распределении через `required_worker_labels` у `Run` (exact-match по заданным ключам).

Операционный runbook и пример запуска удаленных worker описаны в `docs/REMOTE_WORKER_SCALING_V1.md`.

### 23.9 Принцип доступа worker к данным платформы
Для `v2` предпочтительная модель такая:
- source of truth по `Run`, `Match`, `Attempt` и их статусам хранится в `PostgreSQL`;
- `scheduler-service` обновляет эти статусы как управляющий компонент;
- `game-worker` не обязан иметь прямой доступ к `PostgreSQL` и не должен быть зависим от секретов основной БД, если этого можно избежать;
- обмен управляющими сообщениями и техническими событиями между control plane и worker должен идти через `Redis` и/или внутренние сервисные endpoints.

### 23.10 Последовательность распределенного запуска
Нормативная схема запуска:
1. `api-service` создает доменную сущность `Match` или `SingleTaskAttempt`.
2. `scheduler-service` переводит запуск в `queued`, фиксирует snapshot и создает/обновляет `Run`.
3. `scheduler-service` ставит `Run` в очередь (`/internal/runs/{runId}/schedule`).
4. `game-worker` запрашивает задачу (`/internal/workers/pull-next`) и получает lease на `Run`.
5. `game-worker` фиксирует прием запуска в control plane (`/internal/runs/{runId}/accepted`) и после этого переводит `Run` в `running` (`/internal/runs/{runId}/started`).
6. `game-worker` запускает Docker runtime.
7. `game-worker` публикует heartbeat, lifecycle events, `FRAME`, `EVENT`, `STATS`, `MATCH_RESULT`.
8. после завершения worker подтверждает обработку (`/internal/runs/ack-finished`), а control plane закрывает `Run`.

### 23.10.1 Lease TTL и повторная постановка в очередь
Для устойчивости распределенного запуска вводится обязательная lease-модель:
- при выдаче `Run` через `/internal/workers/pull-next` scheduler создает lease с `leased_at` и `worker_id`;
- каждый lease имеет `lease_ttl_seconds` (конфигурируемый параметр scheduler);
- если lease истек, а `Run` все еще активен (`known`), scheduler обязан автоматически снять истекший lease и вернуть `Run` обратно в очередь;
- повторная постановка должна быть дедуплицирована, чтобы один и тот же `Run` не появлялся в очереди несколько раз одновременно;
- подтверждение `/internal/runs/ack-finished` считается валидным только от того worker, у которого актуальный lease;
- `ack` от устаревшего или неактуального lease (`stale ack`) не должен закрывать `Run` и не должен удалять его из активного множества.

### 23.11 Политика отказа worker
Если worker перестал отправлять heartbeat:
- `scheduler-service` помечает worker как unavailable;
- новые `Run` на него не назначаются;
- зависшие запуски помечаются как `failed` или переводятся в requeue по отдельной retry policy;
- source of truth о результате все равно остается в control plane.

### 23.11.1 Retry/backoff для внутренних HTTP вызовов worker
`game-worker` обязан использовать retry/backoff для временных ошибок внутренних API:
- retry применяется к transport-ошибкам и HTTP `5xx`/`408`/`425`/`429`;
- retry не применяется к детерминированным клиентским ошибкам (`4xx`, кроме перечисленных выше);
- стратегия backoff должна быть экспоненциальной с ограничением максимальной задержки;
- параметры retry (`max_attempts`, `base_delay`, `max_delay`) должны быть конфигурируемыми.

## 25. Матчмейкинг

### 25.1 Для тренировочных лобби
Пользовательский lobby-flow единый для всех `multiplayer`-игр.
Политика попадания готового игрока в матч определяется параметрами игры `min_players_per_match` и `max_players_per_match`, а не отдельным типом "маленького" или "большого" лобби.

Базовый алгоритм тренировочного матчмейкинга:
1. Взять готовых участников, которые не находятся в активном батче.
2. Если участников меньше `min_players_per_match`, оставить лобби в фазе `waiting_players`.
3. Если среди готовых участников нет online-viewer, оставить лобби в фазе `waiting_viewer`.
4. Разбить участников на максимальное число матчей так, чтобы размер каждого матча был в диапазоне `min_players_per_match..max_players_per_match`.
5. Создать все run батча параллельно.
6. Сохранить группы run в `last_scheduled_match_groups`.
7. После завершения всех run батча перейти в replay/result период и не запускать новый батч до его окончания.

Если готовых игроков больше `max_players_per_match`, запускается несколько параллельных матчей.
Если после разбиения остается хвост меньше `min_players_per_match`, он остается в очереди до следующего цикла.

Матчмейкинг может расширяться стратегиями выбора состава:
- `random`;
- `skill_based`.

Если выбран `skill_based`, платформа использует внутреннюю скрытую метрику силы команды для конкретной игры. Для `v2` эта метрика определяется как простая эвристика на основе недавних результатов команды: placements, scores, доля побед и экспоненциальное затухание старых результатов. Если данных недостаточно, `skill_based` обязан деградировать к `random`.

### 25.2 Для соревнований
Посев - настраиваемое свойство соревнования:
- `random`;
- `rating`;
- `manual`.

Матчи соревнований тоже запускаются батчами этапа:
- в `single_elimination` все матчи текущего раунда, у которых известны участники, создаются параллельно;
- продвижение в следующий раунд происходит только после terminal-state всех матчей текущего раунда и разрешения tie-break, если он нужен;
- для multi-team match и `top-k` advancement этап может быть не бинарным, поэтому UI обязан показывать rounds/table representation, а не пытаться любой ценой рисовать бинарную сетку.

### 25.3 Командный матчмейкинг
Матчмейкинг работает по командам, а не по отдельным пользователям.

## 29. Frontend SPA

### 29.1 Основные экраны
Основные экраны SPA должны соответствовать верхнему меню и сценариям из раздела `3.4`.

1. `Login`
2. `TasksCatalogPage` - раздел `Задачи`, каталог задач с группировкой по теме и сложности.
3. `SingleTaskRunPage` - запуск задачи, документация, viewer/replay, сравнение с другими.
4. `LobbyListPage` - раздел `Лобби и соревнования`, список открытых лобби, фильтрация по игре, `+` для teacher/admin.
5. `TrainingLobbyPage` - вкладки `Код`, `Лобби`, `Игра`, `Соревнование`, `Архив соревнований`.
6. `GamesPage` - раздел `Игры`, список игр, документация игр, `+` и настройки для teacher/admin.
7. `MatchWatchPage` - контекстный viewer для live/replay, открывается из задачи, лобби, матча или соревнования.
8. `SystemStatusPage` - раздел `Статус системы / общие настройки`, только admin.

Контекстные панели и views, которые не являются отдельными top-level разделами для ученика:
- code workspace внутри `SingleTaskRunPage` или вкладки `Код` в `TrainingLobbyPage`;
- replay inspector внутри задач, лобби, соревнования или `MatchWatchPage`;
- competition bracket внутри вкладки `Соревнование` лобби;
- game source sync и worker control внутри `SystemStatusPage` или admin-view раздела `Игры`.

### 29.2 Компоненты редактора
- CodeMirror-based editor;
- шаблоны кода от игры;
- подсветка Python;
- сохранение по слотам;
- отладочный вывод;
- блокировка редактирования там, где политика соревнования не позволяет менять код.

### 29.3 Match viewer shell
SPA-страница просмотра матча должна:
- показывать метаданные матча;
- держать iframe renderer;
- получать данные матча;
- пересылать безопасные данные в iframe через `postMessage`.
- иметь replay-player (play/pause, step, speed, timeline slider) поверх `replays.frames`.

### 29.4 Быстрый разбор результата (UX flow)
- На `TaskRunPage` для terminal run автоматически подтягивается replay (`frames/events/summary`).
- На `TrainingLobbyPage` матч открывается как пользовательский просмотр игры, без показа raw `run_id` и frame payload ученику.
- В lobby-attached соревновании матч открывается из вкладки `Соревнование` или `Архив соревнований`; debug-инспекторы доступны только учителю/администратору.
- Отдельный top-level каталог replay не показывается ученику; deep-link на replay поддерживается через `MatchWatchPage` для внутренних и преподавательских сценариев.

### 29.5 Админский экран Game Sources (UX flow)
- Список source показывает `source.status` и `last_sync_status` в виде цветных badge.
- При наличии хотя бы одного source в `syncing` экран включает авто-поллинг списка/истории.
- Для выбранного source доступна панель диагностики: `requested_by`, `build_id`, `commit_sha`, `image_digest`, `error_message`.
- Teacher/admin может вручную переключать source между `active` и `disabled`; для `disabled` запуск sync блокируется policy backend.
- На том же экране есть секция `Worker Control` для переключения статуса worker (`online|offline|draining|disabled`) и просмотра емкости.

## 30. Антиплагиат

### 30.1 Область действия
Сравнение выполняется только в рамках одного соревнования.

### 30.2 Запуск
Запускается автоматически при каждом новом snapshot, используемом в соревновании.

### 30.3 MVP-алгоритм
Для `v2` достаточно локальной эвристики:
- нормализация кода;
- token similarity;
- AST similarity;
- строковое сравнение;
- итоговый weighted score.

### 30.4 Реакция системы
Система:
- не наказывает автоматически;
- создает warning teacher/admin;
- показывает пары похожих решений;
- показывает score сходства;
- сохраняет audit trail review.

## 31. Обновление игры и влияние на активные объекты

### 31.1 Если идет соревнование
Синхронизация игры запрещена.

### 31.2 Если идет тренировочный матч
Матч останавливается и помечается как `canceled`, reason-code: `canceled_by_game_update`.

### 31.3 Если есть открытые лобби
Лобби перезапускаются:
- остаются на месте;
- переключаются на новую `GameVersion`;
- сохраняют команды;
- выполняют повторную валидацию команд против новой версии;
- сохраняют `ready` только у команд, которые остаются совместимыми с новой схемой обязательных слотов;
- переводят остальные команды в `not_ready` с понятной причиной в UI.

## 36. Критерии приемки

1. Пользователь может войти через GeekClass JWT и через dev-login в development.
2. Teacher/admin может зарегистрировать игру из публичного git-репозитория и вручную синхронизировать ее.
3. После синхронизации создается новая `GameVersion` с commit SHA и активируется.
4. Teacher/admin может создать тренировочное лобби для встроенной или git-игры.
5. Команда может войти в лобби даже без полного набора кодов, но не может стать `ready`, пока обязательные слоты пусты.
6. Одно лобби может запускать несколько матчей параллельно.
7. Одна команда может одновременно находиться в нескольких лобби.
8. При обновлении игры активные тренировочные матчи останавливаются, а лобби перезапускаются на новой версии с повторной валидацией `ready`-состояния команд.
9. Во время активного соревнования обновление игры запрещено.
10. `single_task` не позволяет одному пользователю иметь два активных запуска одновременно по всей платформе.
11. Платформа в целом поддерживает и `turn_based`, и `script_based` API, а конкретная версия игры явно объявляет один режим `code_api_mode`.
12. `script_based` режим работает без `input()` и под контролем движка по шагам.
13. Renderer сторонней игры работает в `iframe`, не имея прямого доступа к auth tokens.
14. В текущем MVP просмотр live/replay требует сессию; публичный spectator mode появится только как отдельная явная настройка лобби/соревнования.
15. Проверка схожести решений обновляет warning-список teacher/admin в рамках соревнования при появлении новых запусков и при ручном обновлении порога.
16. Есть тесты для доменной логики, критичных API и worker orchestration.

## 38. Инструкция для будущей кодогенерации

При генерации кода обязательно соблюдать:
- DDD-структуру и bounded contexts из этого документа;
- явные доменные инварианты;
- тесты на каждую нетривиальную бизнес-операцию;
- отсутствие бизнес-логики в контроллерах;
- транзакционность критичных операций;
- идемпотентность worker-команд и game events;
- при адаптации legacy-игр: сначала тест, воспроизводящий проблему, затем исправление;
- использование `SVG` как формата по умолчанию для новой графики (игровые ассеты, иконки, схемы), если нет явной причины для raster;
- хорошую читаемость и качество кода вместо "быстрого, но хрупкого" решения.

Если какой-либо участок можно сделать проще без потери требований, выбирать более простую реализацию.

Если какой-либо участок нельзя реализовать без дополнительного решения, по умолчанию использовать:
- безопасное поведение;
- детерминированное поведение;
- минимально сложное поведение.
