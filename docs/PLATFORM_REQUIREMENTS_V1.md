# AI Game Platform - продуктовые и технические требования (v1)

> Этот документ носит обзорный характер. Актуальный основной источник требований для кодогенерации: `docs/SYSTEM_SPEC_V2.md`.

## 1. Назначение документа
Документ фиксирует целевой продукт и архитектуру для превращения текущего движка в платформу для:
- обучения программированию через визуальные задачи;
- проведения соревнований ботов в нескольких форматах;
- масштабирования игровых запусков на несколько серверов.

Документ используется как базовая спецификация для следующих этапов обновления кода.

## 2. Цель продукта
Построить платформу, где участники отправляют код, безопасно запускают его в контейнерах, наблюдают визуализацию задачи или матча и получают прозрачные результаты, рейтинг и турнирные итоги.

## 3. Основные пользовательские сценарии

### 3.1 Одиночные задачи (`single_task`)
Примеры:
- выход из лабиринта;
- сбор монет при движении по карте только вправо и вниз;
- расстановка башен для защиты базы от ботов.

Сценарий:
1. Пользователь пишет код.
2. Пользователь запускает попытку.
3. Платформа выполняет код в изолированном Docker-контейнере.
4. Пользователь видит визуализацию и результат (`win/lose/score`).
5. При необходимости обновляется лидерборд задачи.

### 3.2 Мультиплеерные матчи (`multiplayer`)
Примеры:
- крестики-нолики до 5 на большой доске;
- дуэльные игры;
- танки или free-for-all арены.

Малые и большие лобби больше не являются разными режимами. Любая мультиплеерная игра задает:
- `min_players_per_match`;
- `max_players_per_match`.

Режим тренировки:
- пользователи заходят в лобби;
- матч стартует, когда в очереди есть минимум `min_players_per_match` готовых участников;
- если готовых участников хватает на несколько матчей, платформа запускает несколько матчей параллельно одним батчем;
- следующий батч стартует только после завершения симуляции, replay-периода и короткого result-периода предыдущего батча.

Режим соревнования:
- состав участников фиксируется;
- платформа строит турнирное представление (MVP: single elimination);
- все матчи одного этапа/раунда, у которых известны участники, запускаются параллельно;
- следующий этап формируется после завершения всех матчей предыдущего этапа и применения правил продвижения;
- после завершения доступны итоговая таблица, победитель и история матчей.

## 4. Обязательные функциональные требования

### FR-1. Протокол интеграции игр
Платформа должна поддерживать любое количество игр и задач через единый понятный протокол, без внесения изменений в core-код под каждую новую игру.

Единица интеграции: `game-package`, содержащий:
- `manifest.yaml`;
- игровой раннер;
- описание схемы протокола.

### FR-2. Единая типизация игровых режимов
Каждый интегрируемый модуль обязан объявить один из режимов:
- `single_task`;
- `multiplayer`.

### FR-3. Единый контракт результата
Каждый запуск должен возвращать:
- `status`;
- `winner_ids`;
- `scores`;
- `placements`;
- `replay_ref`;
- `metrics`.

### FR-4. Запуск только в Docker
Весь пользовательский код должен исполняться только в одноразовых Docker-контейнерах с ограничениями ресурсов и изоляцией.

### FR-5. Поддержка распределенных воркеров
Игровые воркеры должны разворачиваться на удаленных серверах.
Control plane распределяет матчи по доступным воркерам по слотам и ресурсам.

### FR-6. Реплеи и наблюдаемость
Для каждого матча и каждой задачи должны быть доступны:
- поток событий и кадров;
- финальный результат;
- логи запуска и детали ошибок;
- трассировка по `match_id`.

## 5. Нефункциональные требования

### NFR-1. Безопасность
- запрет привилегированных контейнеров;
- отсутствие прямых монтирований пользовательского кода в хостовую систему;
- сеть выключена по умолчанию;
- строгие лимиты CPU/RAM/TIME/PIDS.

### NFR-2. Воспроизводимость
- выполнение по `image digest`, а не по изменяемому тегу;
- фиксированная версия протокола;
- поддержка детерминированного `seed`, где это применимо.

### NFR-3. Масштабируемость
- горизонтальное масштабирование через worker-узлы;
- планирование запусков через очередь;
- большое количество параллельных матчей.

### NFR-4. Отказоустойчивость
- heartbeat и проверка живости воркеров;
- политика retry/requeue при падении воркера;
- идемпотентные переходы состояний запуска.

## 6. Целевая архитектура (v1)

### 6.1 Control Plane
- API и аутентификация;
- управление лобби, матчмейкингом и турнирами;
- лидерборды и статистика;
- планировщик и очередь задач.

### 6.2 Worker Plane
- один или несколько узлов `game-worker`;
- каждый воркер забирает задания из очереди;
- каждое задание запускает изолированный Docker-run;
- воркер публикует события протокола и финальный результат.

### 6.3 Транспорт событий
- брокер сообщений для `FRAME/STATS/EVENT/RESULT`;
- веб-сервер транслирует обновления клиентам через обычный WebSocket/SSE в зависимости от реализованного endpoint;
- `Socket.IO` не является частью целевой реализации `v2`.

## 7. Протокол интеграции игр (GIP v1)

### 7.1 Минимальный `manifest`
```yaml
id: ttt_connect5_v1
name: Tic-Tac-Toe Connect 5
version: 1.0.0
protocol_version: 1
mode: multiplayer
min_players_per_match: 2
max_players_per_match: 2
supports_live_join_leave: false
competition_lock_roster: true
scoring_type: winlose
runtime:
  image: registry.example.com/ttt@sha256:...
  language: python
```

Новые manifest-файлы должны использовать только `single_task` или `multiplayer`.
Legacy-значения `small_match`/`massive_lobby` допустимы только для обратной совместимости и должны нормализоваться в `multiplayer`.

### 7.2 События платформа -> игра
- `MATCH_INIT`
- `TICK`
- `PLAYER_JOIN`
- `PLAYER_LEAVE`
- `MATCH_STOP`

### 7.3 События игра -> платформа
- `FRAME`
- `STATS`
- `EVENT`
- `MATCH_RESULT`
- `ERROR`

### 7.4 Статусы выполнения
- `created`
- `queued`
- `running`
- `finished`
- `failed`
- `timeout`
- `aborted`
- `canceled`

## 8. Стратегия хранения игр

Выбранный вариант: **гибридный**.

Этап 1:
- исходный код игр хранится в монорепозитории (`/games/<game_id>`);
- выполнение происходит через Docker-образы по `digest`;
- метаданные и активные версии хранятся в базе данных.

Будущие этапы:
- подключение внешних репозиториев игр;
- сохранение того же протокола и контракта раннера.

## 9. Модернизация фронтенда (Bootstrap 5 + Vue 3)

Текущее состояние:
- frontend реализован как SPA на Vue 3;
- визуальная система построена на Bootstrap 5.

Целевое состояние:
- Bootstrap 5 используется как базовая дизайн-система;
- Vue 3 используется для интерактивных модулей:
  - живые обновления лобби;
  - просмотр состояния матча и реплея;
  - страницы турнирной сетки;
  - панели статуса отправок и запусков.

Итог реализации v2:
1. Старый server-rendered frontend удален из платформенного слоя.
2. Основные пользовательские сценарии перенесены в SPA-маршруты.
3. Реалтайм-обновления лобби/соревнований/запусков работают через унифицированные stream API.
4. Визуализация матчей и реплеев вынесена в отдельные страницы наблюдения.

## 10. Начальный каталог игр (MVP)
- `maze_escape_v1` (`single_task`)
- `coins_right_down_v1` (`single_task`)
- `tower_defense_v1` (`single_task`)
- `ttt_connect5_v1` (`multiplayer`)
- `tanks_ctf_v1` (`multiplayer`)
- графика для референсных примеров готовится в `SVG` по умолчанию.

## 11. План реализации

План этапов A-F реализован в рамках `v2`:
- A: стабилизация базового доменного контура и auth/RBAC;
- B: выделение execution-plane (`scheduler`, `worker`, `builder`) и контейнерный запуск;
- C: тренировки, соревнования, матчмейкинг, лидерборды/результаты;
- D: полный переход на Vue 3 SPA + Bootstrap 5;
- E: референсные игры всех трех режимов с SVG-renderer;
- F: адаптация legacy-игр под v2 API и регрессионное покрытие.

Детальный operational статус фиксируется в:
- `docs/DELIVERY_CHECKLIST_V1.md`;
- `docs/GAME_MIGRATION_STATUS_V1.md`;
- `docs/OPERATIONS_RUNBOOK_V1.md`.

## 12. Критерии приемки v1
1. Новая игра подключается регистрацией пакета и `manifest`, без изменения core-кода оркестрации.
2. Весь пользовательский код запускается только в Docker и соблюдает лимиты выполнения.
3. Платформа умеет одновременно запускать матчи минимум на двух удаленных воркерах.
4. Режим тренировки и режим соревнования по-разному обрабатывают фиксацию состава, как описано в требованиях.
5. Для каждого завершенного запуска доступны реплей, результат и метрики.
6. Существующие игры из `games/` адаптированы под новый API и проходят smoke-тесты без критичных ошибок.

## 13. Ближайшие следующие шаги
1. Уточнять/расширять `SYSTEM_SPEC_V2` при появлении новых game modes и tournament formats.
2. Развивать observability/alerting в целевой инфраструктуре (Prometheus/Grafana/лог-агрегация).
3. При необходимости добавлять новые runtime profiles и языки поверх текущего Python-first контура.
4. Постепенно расширять каталог игр и автопроверок, сохраняя совместимость с v2 протоколом.
