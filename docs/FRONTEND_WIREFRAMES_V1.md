# AI Game Platform - wireframes главных сценарных экранов (v1)

## 1. Назначение документа
Этот документ описывает wireframe-уровень 5 ключевых экранов платформы и дополняет:
- [docs/SYSTEM_SPEC_V2.md](/Users/roctbb/PycharmProjects/ai-game-engine/docs/SYSTEM_SPEC_V2.md)
- [docs/FRONTEND_VISUAL_BRIEF_V1.md](/Users/roctbb/PycharmProjects/ai-game-engine/docs/FRONTEND_VISUAL_BRIEF_V1.md)

Документ нужен для:
- проектирования Vue 3 SPA;
- сборки page layout;
- проектирования компонентной библиотеки;
- последующей генерации экранов без импровизации в архитектуре UI.

Перед использованием wireframes обязательно сверяться с пользовательскими сценариями из `docs/SYSTEM_SPEC_V2.md`, раздел `3.4`.

Wireframes описывают возможную форму экранов, но не являются самостоятельным источником навигационной логики. Если текущий layout мешает сценарию пользователя, приоритет имеет сценарий:
- студент решает задачу;
- студент участвует в лобби через вкладки `Код`, `Лобби`, `Игра`;
- студент участвует в соревновании;
- зритель смотрит live/replay;
- teacher проводит лобби или соревнование;
- teacher/admin синхронизирует игру;
- admin управляет worker plane.

## 2. Какие экраны считаются главными
Для первой полноценной визуальной реализации главными считаются экраны, соответствующие верхнему меню и ключевым сценариям:

1. Каталог задач
2. `single_task` run page
3. Список лобби и соревнований
4. Тренировочное лобби с вкладками `Код`, `Лобби`, `Игра`, `Соревнование`, `Архив соревнований`
5. Игры
6. Статус системы / общие настройки для admin
7. Match viewer как контекстный экран просмотра

Именно эти экраны должны стать ядром SPA и определять структуру frontend. Рабочее место кода и сетка соревнования являются важными views, но для ученика они живут внутри задачи или лобби, а не как отдельные top-level разделы.

## 3. Общие правила wireframe

### 3.1 Общий shell
Все 5 экранов живут внутри единого `AppShell`:
- `AppSidebar`
- `TopBar`
- `ContentArea`

Desktop:
- sidebar слева;
- top bar сверху;
- основной контент в центре;
- contextual rail справа там, где нужно.

Mobile:
- top bar сверху;
- drawer вместо sidebar;
- bottom navigation;
- длинные страницы разбиваются на stacked sections;
- боковые панели превращаются в tabs, drawers или bottom sheets.

### 3.1.1 Верхнее меню

Основные пункты верхнего меню:
- `Задачи`;
- `Лобби и соревнования`;
- `Игры`;
- `Статус системы / общие настройки` - только для admin.

Role-specific actions:
- teacher/admin видит `+` в разделе `Лобби и соревнования` для создания лобби;
- teacher/admin видит `+` в разделе `Игры` для добавления игры;
- teacher/admin может менять настройки игр из раздела `Игры`;
- ученик не видит отдельные пункты `Команды`, `Матчи`, `Повторы`, `Администрирование`.

### 3.2 Общее обозначение блоков
В wireframe используются такие обозначения:
- `Header` — главный заголовок страницы
- `Toolbar` — панель быстрых действий
- `Filters` — фильтры/поиск/переключатели
- `Main` — центральное содержимое страницы
- `Rail` — боковая панель
- `Panel` — логический контейнер
- `List` — список карточек/элементов
- `Viewer` — игровая визуализация
- `Editor` — редактор кода

### 3.3 Подход к desktop layout
Для desktop предполагается условная сетка:
- sidebar: 240-280 px
- основной контент: fluid
- right rail: 320-400 px при необходимости

### 3.4 Подход к mobile layout
Для mobile:
- один главный вертикальный поток;
- sticky CTA, если действие критично;
- editor/viewer/логи не должны одновременно пытаться жить в трех колонках.

## 5. Контекстный view - Код игрока
Маршрут:
- `/games/:gameId/workspace` или `/teams/:teamId`

Цель:
- управлять слотами/ролями команды;
- редактировать код по слотам;
- понимать состояние кода команды;
- понимать, какие snapshot будут зафиксированы, если новый запуск создать сейчас.

На текущем этапе пользователь не выбирает одну из нескольких команд для игры. Его игрок для игры - это canonical набор кодов по ролям; технически он может храниться как `Team`, но в UI показывается как игрок/участник. Этот view обычно встроен в `SingleTaskRunPage` или вкладку `Код` в `TrainingLobbyPage`.

### 5.5 Критичные состояния
- обязательный слот пуст;
- слот заполнен, но код изменен после последнего snapshot;
- политика соревнования блокирует редактирование;
- пользователь не может встать в очередь;
- игра обновилась и слот стал несовместимым.

### 5.6 Доменные правила для этого экрана
- `ready` не является статусом слота и не должен так отображаться в slot rail;
- допустимые slot-state в UI: `filled`, `empty`, `dirty`, `locked`, `incompatible`;
- для student-flow не показывать выбор команды, пока действует правило "одна команда пользователя на игру";
- если пользователь попал сюда из лобби, сохранение кода во время ожидания или текущей игры должно объяснять, что изменения применятся к следующей игре;
- экран не обещает "следующий матч" как глобальный факт, потому что команда может участвовать в нескольких лобби;
- допустима только preview-формулировка: какие snapshot будут созданы, если новый запуск инициировать сейчас.

## 6. Экран 3 - `single_task` run page
Маршрут:
- `/tasks/:gameId/run`

Цель:
- дать участнику полноценное пространство для запуска, наблюдения и улучшения решения.

### 6.0 Source of truth для кода на странице `single_task`
Для `single_task` run page не существует отдельного второго хранилища кода.

Экран `/tasks/:gameId/run` обязан работать поверх canonical workspace той же однослотовой персональной команды пользователя для этой игры:
- source of truth остается `TeamSlotCode`;
- попытка `SingleTaskAttempt` использует snapshot этого canonical кода в момент перехода `created -> queued`;
- встроенный редактор на run page редактирует тот же самый код, что и специализированный workspace команды;
- если у пользователя еще нет персональной однослотовой команды для этой игры, платформа может создать ее автоматически как технический workspace.

Из этого следует:
- run page - это специализированный execution shell над тем же кодовым workspace;
- нельзя проектировать run page как отдельный независимый редактор с собственной моделью хранения.

- если активная попытка уже существует, запуск блокируется на уровне UX и API.

## 7. Экран 4 - Тренировочное лобби
Маршрут:
- `/lobbies/:lobbyId`

Цель:
- дать пользователю единый сценарий участия в лобби;
- дать заполнить код по ролям;
- показать участников по группам: играют, готовятся, в очереди;
- показать текущие матчи и архив матчей;
- показать текущую игру пользователя.
- поддержать соревнование, прикрепленное к этому лобби.

Один и тот же layout используется для `small_match` и `massive_lobby`.
В `small_match` игрок после `Готов / Play` может ждать подбора матча в очереди.
В `massive_lobby` готовый игрок может сразу попадать в текущую игру/мир.

Для student-flow лобби является основным контейнером из трех вкладок:
- `Код` - canonical редактор ролей для игры и документация игры/задачи;
- `Лобби` - участники по группам, очередь, текущие матчи, архив матчей;
- `Игра` - текущая игра пользователя.

Дополнительные вкладки появляются по состоянию:
- `Соревнование` - только пока в лобби идет активное соревнование или оно ожидает ручного завершения teacher/admin;
- `Архив соревнований` - если в лобби было хотя бы одно завершенное соревнование.

Активная вкладка выбирается ситуационно: незаполненные роли ведут в `Код`, ожидание матча - в `Лобби`, найденная игра пользователя - в `Игра`, активное соревнование - в `Соревнование`.

### 7.5 Критичные состояния
- обязательные слоты пусты;
- игрок blocked;
- пользователь уже в очереди;
- пользователь уже в текущей игре;
- у пользователя нет текущей игры для вкладки `Игра`;
- в лобби идет соревнование и вход/выход заблокирован;
- активное соревнование завершило все игры, но teacher/admin еще не нажал `Завершить соревнование`;
- лобби обновляется после game sync;
- матчмейкинг временно недоступен;
- spectator mode выключен;
- лобби `draft`, `paused`, `updating` или `closed`, и часть действий недоступна.

### 7.6 Доменные правила для этого экрана
- lifecycle-state лобби должен быть видимым всегда, а не только через disabled buttons;
- при первом входе пользователя в лобби платформа создает запись участия и обеспечивает наличие команды пользователя для игры этого лобби;
- вкладка `Код` обязана содержать доступ к документации игры/задачи как части текущей игры (`player_guide` и/или `sdk_guide`, если они есть в игровом пакете);
- доступность действия `Готов` / `Play` определяется одновременно статусом лобби и заполненностью обязательных ролей;
- действие `Не готов` / `Stop` выводит пользователя из очереди или текущей игры и переводит его в группу `готовятся`;
- состояние `updating` должно явно объяснять пользователю, что идет переключение версии игры и повторная валидация `ready`;
- во время текущей игры пользователь может вернуться во вкладку `Код` и сохранить изменения для следующей игры;
- если пользователь меняет код и нажимает `Готов` / `Play`, изменения применяются со следующей игры;
- вкладка `Лобби` обязана показывать участников группами `играют`, `готовятся`, `в очереди`;
- по каждому игроку нужно показывать доступную агрегированную статистику: число побед, среднее число очков, недавнюю форму или другой показатель силы;
- лобби должно помогать игроку понять, с кем он играет и насколько хорошо выступают эти участники;
- вкладка `Лобби` обязана показывать текущие матчи и архив матчей этого лобби;
- открытие текущего или архивного матча запускает match viewer в отдельной вкладке/странице;
- вкладка `Игра` показывает текущую игру пользователя двухколоночно: слева визуализация, справа статистика игроков.
- teacher/admin может нажать `Начать соревнование` из лобби, выбрать настройки и запустить соревнование;
- во время соревнования лобби показывает persistent banner `Идет соревнование`;
- во время соревнования из лобби нельзя выйти и в него нельзя войти новым участникам;
- вкладка `Соревнование` показывает realtime-сетку: кто играет, кто победил, кто проиграл;
- клик по матчу во вкладке `Соревнование` открывает match viewer;
- текущая игра пользователя во время соревнования все равно показывается во вкладке `Игра`;
- вкладка `Соревнование` пропадает после того, как все игры завершены, победитель выбран и teacher/admin нажал `Завершить соревнование`;
- вкладка `Архив соревнований` появляется после первого завершенного соревнования в лобби и дает доступ к итогам прошлых соревнований.

## 8. Контекстный view - Соревнование
Маршрут:
- вкладка `Соревнование` внутри `/lobbies/:lobbyId`
- `/competitions/:competitionId` допустим как admin/deep-link view, но не как основной student-flow

Цель:
- показать структуру турнира;
- показать progression;
- дать быстрый переход к матчам;
- поддержать модерацию и spectator mode.

Для основного student-flow соревнование прикреплено к лобби и открывается как вкладка `Соревнование`.

### 8.1.1 Wireframe для multi-team elimination node
Если формат соревнования остается elimination, но матч содержит `N > 2` команд или правило продвижения равно `top-k`, бинарный узел сетки использовать нельзя.

В таком случае node обязан выглядеть как match-card внутри сетки:

```text
+--------------------------------------+
| Матч 18                              |
| Игрок A                              |
| Игрок B                              |
| Игрок C                              |
| Игрок D                              |
|--------------------------------------|
| Проходят дальше: 2                   |
| 1. Игрок C                           |
| 2. Игрок A                           |
| Ничья: ручное решение                |
+--------------------------------------+
```

Такой node должен явно показывать:
- список всех участников матча;
- правило продвижения;
- фактически прошедшие дальше команды;
- если нужно, наличие ничьей и ручного решения.

### 8.1.2 Когда bracket нельзя делать главным представлением
Если elimination-логика не читается естественно как bracket из-за:
- слишком большого числа команд в одном матче;
- `top-k` продвижения;
- частых неоднозначных tie-break;

то вкладка `Соревнование` должна по умолчанию открываться не в bracket view, а в `Rounds / Matches / Standings` представлении.

Bracket в этом случае может оставаться дополнительным режимом просмотра, но не единственным и не главным.

### 8.6 Критичные состояния
- турнир еще не стартовал;

- есть banned team;
- матч завершен, но результат невалиден;

- часть матчей canceled или aborted.

### 8.7 Доменные правила для этого экрана

## 10. Приоритет реализации
С точки зрения UX и ценности я бы реализовывал эти wireframes в таком порядке:

1. `AppShell` и верхнее меню
2. `TasksCatalogPage`
3. `SingleTaskRunPage`
4. `LobbyListPage`
5. `TrainingLobbyPage` с вкладками `Код`, `Лобби`, `Игра`
6. Вкладка `Соревнование` и `Архив соревнований` внутри лобби
7. `GamesPage`
8. `MatchWatchPage`
9. `SystemStatusPage` для admin

Причина:
- верхнее меню и каталог задач задают понятную точку входа;
- экран задачи и лобби определяют основные student-сценарии;
- соревнование является режимом лобби, поэтому строится поверх уже понятного lobby shell;
- системные экраны важны, но не должны определять опыт ученика.

## 11. Инструкция для дальнейшей генерации
При последующей генерации frontend-кода:
- не отходить от этих зон страницы без явной причины;
- держать главный CTA визуально приоритетным;
- не заменять layout таблицей, если wireframe предполагает panel/card composition;
- все 5 экранов делать сначала для desktop и mobile, а не только desktop;
- viewer и editor рассматривать как first-class layout blocks, а не как случайные вложенные widgets.
